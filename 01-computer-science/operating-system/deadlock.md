# 데드락 (Deadlock)

> **한 줄 정리**
> 두 개 이상의 실행 흐름이 **서로 상대가 점유한 자원을 기다리며 무한히 멈춰있는 상태**.
> 4가지 조건이 **모두 만족**될 때만 발생한다. 하나만 깨도 데드락은 발생하지 않는다.

---

## 1. 데드락이란?

### 1.1 정의
프로세스/스레드들이 서로 **상대가 가진 자원을 기다리며** 더 이상 진행하지 못하는 상태.

### 1.2 가장 단순한 예시
```
스레드 A: lock(R1) → lock(R2) 시도 → 대기 (R2는 B가 점유 중)
스레드 B: lock(R2) → lock(R1) 시도 → 대기 (R1은 A가 점유 중)

결과: 둘 다 영원히 대기 → 데드락
```

**식사하는 철학자 문제 (Dining Philosophers)**
- 원탁에 철학자 5명, 사이마다 포크 1개씩 (총 5개)
- 식사하려면 양쪽 포크 2개가 필요
- 모두 동시에 왼쪽 포크를 잡으면 → 오른쪽 포크는 영원히 비지 않음 → 데드락

### 1.3 데드락 vs 기아 vs 라이브락

| 구분 | 설명 |
|------|------|
| **데드락 (Deadlock)** | 자원을 서로 기다리며 모두 멈춤 |
| **기아 (Starvation)** | 특정 스레드만 계속 자원을 얻지 못함 (다른 건 진행 중) |
| **라이브락 (Livelock)** | 스레드들이 계속 상태를 바꾸지만 **실질적 진행 없음** (서로 양보만 반복) |

## 2. 데드락의 4가지 필요 조건 (Coffman Conditions, 1971)

**4가지가 모두 만족**되어야 데드락이 발생할 수 있다.
=> 거꾸로, **하나만 깨도 데드락은 발생하지 않는다**.

### 2.1 상호 배제 (Mutual Exclusion)
> 자원은 한 번에 한 스레드만 사용 가능.

- 공유 가능한 자원(read-only 파일 등)은 데드락 안 일어남
- 프린터, 락 같은 비공유 자원이 대상

### 2.2 점유와 대기 (Hold and Wait)
> 이미 자원을 가진 채로 다른 자원을 추가로 요청하며 대기.

- 자원 하나만 들고 또 다른 걸 기다리는 상태

### 2.3 비선점 (No Preemption)
> 자원을 강제로 빼앗을 수 없음. 보유자가 자발적으로 반환해야 함.

- CPU는 선점 가능 → 데드락 X
- 락, DB 트랜잭션 등은 보통 비선점

### 2.4 순환 대기 (Circular Wait)
> 대기 그래프에 **사이클**이 존재.

```
T1 → (R2 대기) → T2 → (R3 대기) → T3 → (R1 대기) → T1
```

**자원 할당 그래프 (Resource Allocation Graph)**
- 노드: 프로세스(P), 자원(R)
- 엣지: `P → R` (요청), `R → P` (할당)
- **사이클 존재 ⇒ 데드락 가능성**
- 자원이 단일 인스턴스면 사이클 = 데드락 확정

## 3. 데드락 해결 전략 4가지

| 전략 | 시점 | 핵심 아이디어 |
|------|------|---------------|
| **예방 (Prevention)** | 발생 전 | 4가지 조건 중 하나를 사전에 막음 |
| **회피 (Avoidance)** | 할당 시 | 안전 상태일 때만 할당 (Banker's Algorithm) |
| **탐지 & 복구 (Detection & Recovery)** | 발생 후 | 주기적으로 검사 후 복구 |
| **무시 (Ignore)** | - | 드물면 그냥 무시 (Ostrich Algorithm) |

---

## 4. 데드락 예방 (Prevention)

4가지 조건을 **사전에 차단**.

### 4.1 상호 배제 깨기
- 자원을 **공유 가능하게** 설계 (read-only, 불변 데이터)
- 모든 자원에 적용 불가능 (어차피 락이 필요한 자원이 있음)

### 4.2 점유와 대기 깨기
- 스레드가 **시작 시 필요한 모든 자원을 한꺼번에 요청**
- 단점: 자원 활용도↓ (오래 점유), 기아 가능성

### 4.3 비선점 깨기
- 자원을 못 얻으면 **보유 중인 자원을 모두 반납**하고 다시 요청
- 단점: 작업 상태 저장/복원 비용 큼. 락 같은 자원에는 적용 어려움

### 4.4 순환 대기 깨기 (가장 실용적)
- **자원에 전역 순서를 부여**하고, **오름차순으로만 락 획득**
- 예: 모든 스레드가 `lock(A) → lock(B) → lock(C)` 순서를 지키면 사이클 불가능
- **실무에서 가장 많이 쓰이는 예방법**

```c
// 잘못된 예
thread1: lock(A); lock(B);
thread2: lock(B); lock(A);   // 데드락 가능

// 올바른 예
thread1: lock(A); lock(B);
thread2: lock(A); lock(B);   // 항상 A → B
```

## 5. 데드락 회피 (Avoidance) — Banker's Algorithm

### 5.1 아이디어
자원 요청이 들어왔을 때, **할당 후에도 안전 상태(Safe State)인지** 미리 검사. 안전하면 할당, 아니면 대기.

### 5.2 안전 상태(Safe State)
모든 프로세스가 **어떤 순서로든 끝까지 완료 가능한 자원 할당 시퀀스가 존재**하는 상태.

### 5.3 필요한 자료구조
| 자료 | 의미 |
|------|------|
| `Available[m]` | 가용 자원 |
| `Max[n][m]` | 각 프로세스가 최대 요청할 자원 |
| `Allocation[n][m]` | 현재 할당된 자원 |
| `Need[n][m]` | 추가로 필요한 자원 (`Max - Allocation`) |

### 5.4 예시
프로세스 5개, 자원 3종류(A, B, C)
```
Total: A=10, B=5, C=7

       Max         Allocation     Need
P0  (7,5,3)        (0,1,0)       (7,4,3)
P1  (3,2,2)        (2,0,0)       (1,2,2)
P2  (9,0,2)        (3,0,2)       (6,0,0)
P3  (2,2,2)        (2,1,1)       (0,1,1)
P4  (4,3,3)        (0,0,2)       (4,3,1)

Available = (3,3,2)

Safe sequence: P1 → P3 → P4 → P2 → P0 ⇒ 안전 상태
```

### 5.5 한계
- 모든 프로세스의 **최대 자원 요구량을 사전에 알아야 함**
- 프로세스가 동적으로 생기는 환경에서 적용 어려움
- 실무에서는 거의 안 쓰임 (교과서적 개념)

## 6. 데드락 탐지 & 복구

### 6.1 탐지 (Detection)
- 자원 할당 그래프에서 **사이클 검출**
- 단일 인스턴스 자원: 사이클 = 데드락
- 다중 인스턴스 자원: Banker's와 유사한 탐지 알고리즘 필요
- 검사 주기는 트레이드오프 (자주 → 오버헤드, 드물게 → 발견 지연)

### 6.2 복구 (Recovery)
**프로세스 종료**
- 모든 데드락 프로세스 일괄 종료 (간단하지만 손실 큼)
- 사이클이 사라질 때까지 하나씩 종료 (선택 기준: 우선순위, 진행도, 자원 사용량)

**자원 선점**
- 데드락 프로세스로부터 자원을 빼앗아 다른 프로세스에 할당
- **롤백 필요**: 빼앗긴 프로세스는 안전한 체크포인트로 복귀
- **기아 방지**: 같은 프로세스만 계속 희생되지 않도록

## 7. 데드락 무시 (Ostrich Algorithm)

"타조 알고리즘" — 머리를 모래에 묻듯 데드락을 그냥 무시.

- **현실에서 가장 많이 쓰이는 전략**
- 데드락 발생 빈도가 낮고, 예방/탐지 비용이 더 큰 경우 합리적
- Linux, Windows 같은 범용 OS의 기본 자세
- 발생하면 사용자가 알아서 재시작/Kill

## 8. 실무에서 데드락 피하기

### 8.1 락 순서 일관성 (Lock Ordering)
**가장 중요한 원칙.** 모든 코드 경로에서 락을 항상 같은 순서로 획득.

```java
// 안전: 항상 작은 ID → 큰 ID 순서
Account a = ..., b = ...;
Account first  = a.id < b.id ? a : b;
Account second = a.id < b.id ? b : a;

synchronized (first) {
    synchronized (second) {
        transfer(a, b, amount);
    }
}
```

### 8.2 타임아웃 (tryLock with timeout)
일정 시간 안에 락을 얻지 못하면 포기 + 재시도.

```java
if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
    try {
        // critical section
    } finally {
        lock.unlock();
    }
} else {
    // 재시도 또는 다른 전략
}
```

### 8.3 락 범위 최소화
- 임계 구역을 짧게
- 락 안에서 **I/O, 네트워크, 외부 API 호출 금지** (블로킹 → 데드락 위험 증가)
- 락 안에서 다른 락 획득 자제 (어쩔 수 없으면 순서 일관성 유지)

### 8.4 계층적 락 (Lock Hierarchy)
- 락에 레벨을 부여하고 낮은 레벨 → 높은 레벨 순서로만 획득
- 같은 레벨끼리는 동시에 잡지 않음

### 8.5 락 분할 (Lock Striping)
- 큰 자료구조를 여러 락으로 분할
- 예: `ConcurrentHashMap`은 내부적으로 여러 세그먼트 락 사용

### 8.6 불변 객체 / 메시지 패싱
- 공유 상태 자체를 줄임
- Actor 모델, Go의 채널 등

## 9. DB 데드락

데드락은 OS만의 문제가 아니다. **DB 트랜잭션에서도 자주 발생.**

### 9.1 시나리오
```sql
-- 트랜잭션 A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- row 1 lock
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- row 2 대기

-- 트랜잭션 B (동시 실행)
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;   -- row 2 lock
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- row 1 대기 → 데드락
```

### 9.2 DB의 데드락 처리
- 대부분의 RDBMS는 **데드락 탐지기**를 내장 (대기 그래프 사이클 검출)
- 발견 시 **희생자(Victim) 트랜잭션을 자동 롤백**
- 애플리케이션은 **데드락 예외를 받아 재시도** 로직 필요

**MySQL/PostgreSQL의 데드락 메시지**
```
ERROR 1213 (40001): Deadlock found when trying to get lock;
try restarting transaction
```

### 9.3 DB 데드락 줄이는 방법
- 트랜잭션을 짧게 유지
- 인덱스 잘 만들기 (불필요한 행 잠금 방지)
- 같은 순서로 행 접근 (`ORDER BY id`)
- 적절한 격리 수준 선택
- `SELECT ... FOR UPDATE`의 범위를 최소화

## 10. 면접 단골 질문

### Q1. 데드락의 4가지 필요 조건은?
→ §2 (Mutual Exclusion / Hold and Wait / No Preemption / Circular Wait)
→ 모두 만족해야 발생. **하나만 깨도 예방 가능.**

### Q2. 가장 실용적인 데드락 예방법은?
**락 순서 일관성 (Circular Wait 깨기).** 자원에 전역 순서를 부여하고 그 순서로만 획득.

### Q3. 데드락과 라이브락의 차이는?
- 데드락: **멈춤**. 아무 진행 없음.
- 라이브락: **계속 움직이지만 진행 없음**. (예: 서로 양보만 반복)

### Q4. Banker's Algorithm의 한계는?
- 최대 자원 요구량을 미리 알아야 함
- 프로세스 수, 자원 수에 따라 비용 ↑
- 동적 환경에서 비현실적 → 실무에선 거의 안 씀

### Q5. 운영체제가 데드락을 어떻게 처리하나?
- Linux, Windows 같은 범용 OS는 **무시(Ostrich)** 전략
- 대신 사용자/관리자가 프로세스 강제 종료
- DB나 분산 시스템은 **탐지 & 복구** 전략을 적극 사용

### Q6. tryLock과 lock의 차이는? 데드락 방지에 어떻게 쓰이나?
- `lock`: 무한 대기
- `tryLock`: 즉시 실패 또는 타임아웃 후 실패
- 데드락이 의심되는 상황에서 `tryLock` + 백오프 + 재시도로 회피 가능

### Q7. 식사하는 철학자 문제를 어떻게 해결하나?
- **자원 순서**: 포크에 번호를 부여하고 작은 번호 → 큰 번호 순서로만 집기
- **자원 제한**: 동시에 식사할 수 있는 철학자 수를 N-1로 제한 (세마포어)
- **비대칭**: 짝수 번호는 왼쪽 먼저, 홀수 번호는 오른쪽 먼저
- **중재자**: 한 번에 한 명만 포크 집을 수 있도록
