# I/O 관리 (I/O Management)

> **한 줄 정리**
> 다양한 I/O 장치(키보드, 디스크, 네트워크 카드, GPU...)를 **통일된 인터페이스**로 추상화하고,
> CPU가 I/O를 기다리며 놀지 않도록 **인터럽트 / DMA / 버퍼링 / 캐싱**을 활용하는 일.

---

## 1. 왜 I/O 관리가 어려운가?

### 1.1 장치들의 다양성
- 키보드 (수십 바이트/초), SSD (수 GB/초), 네트워크 (10/100Gbps), GPU (수백 GB/초)
- 전송 단위: 바이트 vs 블록
- 접근 방식: 순차 vs 랜덤
- 동기 vs 비동기

### 1.2 속도 격차
- CPU: ~ns
- RAM: ~10 ns
- SSD: ~50 μs
- HDD: ~10 ms (CPU의 약 10⁷배 느림)
- 네트워크: 수 ms ~ 수백 ms

→ CPU는 I/O보다 **수만~수억 배 빠름**. 잘못 설계하면 CPU가 I/O 기다리다 놀게 됨.

### 1.3 핵심 문제
- CPU를 I/O 대기에서 해방시키기
- 다양한 장치를 통일된 인터페이스로
- 안정성 (장치 오류 처리)
- 동시 접근 관리

---

## 2. I/O 장치 분류

### 2.1 전송 단위별

| 종류 | 단위 | 예시 |
|------|------|------|
| **Character Device (문자 장치)** | 바이트 스트림 | 키보드, 마우스, 시리얼 포트, 터미널 |
| **Block Device (블록 장치)** | 고정 크기 블록 | HDD, SSD, USB 메모리 |
| **Network Device** | 패킷 | NIC (Ethernet, Wi-Fi) |

### 2.2 동작 방식별
- **Programmed I/O (PIO)**: CPU가 직접 데이터를 전송
- **Interrupt-driven I/O**: 장치가 준비되면 인터럽트로 통지
- **DMA (Direct Memory Access)**: 장치가 CPU 개입 없이 메모리와 직접 데이터 전송

---

## 3. I/O 방식의 진화

### 3.1 폴링 (Programmed I/O)
```
loop:
  if (device_ready) {
    read_data();
    process();
  }
```
- CPU가 장치 상태를 계속 확인 → **Busy Waiting**
- CPU 100% 점유하면서도 다른 일은 못 함
- 임베디드/RTOS에서 짧은 반응시간이 필요할 때 사용

### 3.2 인터럽트 기반 I/O
```
1. CPU: I/O 요청 후 다른 작업 진행
2. 장치: 작업 완료 → 인터럽트
3. CPU: ISR 실행 → 데이터 처리
```
- CPU 활용도 ↑
- 단점: 데이터 1바이트마다 인터럽트면 오히려 비효율 (인터럽트 처리 비용)

### 3.3 DMA (Direct Memory Access)
> **DMA 컨트롤러**가 CPU 대신 장치 ↔ 메모리 데이터 전송. CPU는 시작/완료만 관여.

```
CPU → DMA 컨트롤러에게 (소스, 목적지, 크기) 알림
       ↓
DMA가 직접 메모리에 데이터 복사
       ↓
완료 시 CPU에 인터럽트 1회
```

**장점**:
- 데이터 한 블록 전체에 인터럽트 1번만
- CPU는 그 시간에 다른 일

**Cycle Stealing**: DMA가 메모리 버스를 잠시 점유 → CPU의 메모리 접근이 살짝 느려질 수 있음. 그래도 인터럽트 N번보다 훨씬 효율.

**사용**: 디스크, 네트워크, GPU 등 **블록 단위 전송**이 큰 장치 거의 모두

## 4. I/O 소프트웨어 계층

```
┌─────────────────────────┐
│  사용자 프로세스         │   open/read/write 호출
├─────────────────────────┤
│  장치 독립적 I/O 소프트웨어│   버퍼링, 캐싱, 명명, 보호
├─────────────────────────┤
│  디바이스 드라이버        │   장치별 코드
├─────────────────────────┤
│  인터럽트 핸들러         │   하드웨어 신호 처리
├─────────────────────────┤
│   하드웨어 (장치)        │
└─────────────────────────┘
```

각 계층은 아래 계층의 디테일을 숨기고 위로는 단순한 인터페이스를 제공.

### 4.1 디바이스 드라이버
- 장치마다 고유한 명령/레지스터를 추상화
- OS가 정의한 표준 인터페이스(open/read/write/ioctl) 구현
- 보통 커널 모듈로 로드/언로드 (Linux의 `insmod`/`rmmod`)

### 4.2 디바이스 파일 (Unix)
- `/dev/sda`, `/dev/tty`, `/dev/null` 등
- 파일처럼 보이지만 실제론 드라이버를 가리킴
- `ls -l /dev`에서 첫 글자가 `b`(block) 또는 `c`(character)

## 5. 버퍼링 (Buffering)

### 5.1 왜 필요한가?
- 생산자와 소비자 속도 차이 흡수
- 큰 단위로 모아서 한 번에 전송 (시스템 콜 비용 절감)
- 데이터를 재시도해야 할 수 있음 (네트워크 등)

### 5.2 종류

| 종류 | 설명 |
|------|------|
| **단일 버퍼** | 하나의 버퍼만 사용. 채우는 동안 처리 못 함 |
| **이중 버퍼 (Double Buffer)** | 하나는 채우고 하나는 처리 → 병렬화 |
| **순환 버퍼 (Circular Buffer / Ring Buffer)** | 여러 칸의 원형 큐. 생산자-소비자 비동기화 |

### 5.3 캐싱과의 차이
- **버퍼**: 단위 변환, 속도 격차 흡수가 목적 (한 번 쓰고 버림)
- **캐시**: 자주 쓰는 데이터를 가까이 두기 (반복 사용)
- 페이지 캐시 같은 경우는 사실상 버퍼 + 캐시 역할 동시

### 5.4 표준 라이브러리의 버퍼링
- `printf`는 줄 단위/블록 단위로 버퍼링
- 그래서 segfault 직전 `printf` 출력이 안 보이는 경우 있음 → `fflush()` 또는 `\n` 필요

## 6. I/O 스케줄링

디스크에 여러 요청이 동시에 오면 어떤 순서로 처리할지 결정.
**HDD에서는 매우 중요했음**. SSD는 랜덤 접근이 빨라서 영향이 적어졌지만 여전히 의미 있음.

### 6.1 알고리즘

#### FCFS
- 들어온 순서대로
- 단순하지만 헤드 이동 거리 ↑

#### SSTF (Shortest Seek Time First)
- 현재 헤드 위치에서 가장 가까운 요청부터
- 평균 응답시간 ↓
- 멀리 있는 요청은 **기아** 가능

#### SCAN (엘리베이터 알고리즘)
- 헤드가 한 방향으로 끝까지 가면서 처리 → 반대로 → 반복
- 기아 없음, 비교적 공정

#### C-SCAN (Circular SCAN)
- 한 방향으로만 처리, 끝나면 처음으로 점프
- 응답 시간 분포가 더 균일

#### LOOK / C-LOOK
- SCAN/C-SCAN 변형. 끝까지가 아니라 **요청 있는 곳까지만** 이동

### 6.2 Linux의 I/O 스케줄러
- **noop**: 단순 FIFO + 머지 (SSD에 적합)
- **deadline**: 각 요청에 마감 시한 부여 → 기아 방지
- **CFQ (Completely Fair Queuing)**: 프로세스별 큐로 공정 분배
- **mq-deadline / kyber / bfq**: 멀티큐 시대의 모던 스케줄러
- 확인: `cat /sys/block/sda/queue/scheduler`

### 6.3 SSD 시대의 변화
- HDD: 탐색 시간(seek) 지배적 → 스케줄링이 큰 효과
- SSD: 위치 무관 빠름 → **스케줄러보다 큐잉/병렬화가 더 중요**
- NVMe는 수십 개의 큐를 활용 → 멀티큐 스케줄러 필요

## 7. 메모리 매핑 I/O (Memory-Mapped I/O, MMIO)

### 7.1 두 가지 I/O 방식
- **Port-Mapped I/O (PMIO)**: 전용 명령어(in/out) + 별도 주소 공간
- **Memory-Mapped I/O (MMIO)**: 장치 레지스터를 메모리 주소 공간에 매핑

x86은 둘 다 지원하지만 현대 시스템은 거의 **MMIO** 사용. 명령어 통일 + 64비트 큰 주소 공간 활용.

### 7.2 mmap (파일 매핑)
- `mmap()`으로 파일을 메모리에 매핑하면 그 영역을 메모리 접근하듯 읽기/쓰기 가능
- 페이지 폴트 시 디스크에서 자동 로드 (디맨드 페이징)
- 큰 파일을 효율적으로 다룰 때 유용 (DB, 대용량 로그)

---

## 8. 비동기 I/O & 폴링 기반 멀티플렉싱

### 8.1 동기 vs 비동기
- **동기 (Blocking)**: `read()`가 데이터 올 때까지 멈춤
- **비동기 (Non-blocking)**: `read()`가 즉시 반환 (있으면 데이터, 없으면 EAGAIN)
- **비동기 알림**: 완료 시 콜백/이벤트

### 8.2 I/O 멀티플렉싱
한 스레드에서 여러 fd를 동시에 감시.

| API | 특징 |
|-----|------|
| **select** | 가장 오래됨. fd 개수 제한 (1024). O(N) 스캔 |
| **poll** | fd 개수 제한 없음. O(N) 스캔 |
| **epoll (Linux)** | O(1)에 가까운 효율, 이벤트 등록 방식. 고성능 서버의 표준 |
| **kqueue (BSD/macOS)** | epoll과 유사 |
| **IOCP (Windows)** | Completion Port 모델 |

### 8.3 io_uring
- Linux 5.1+ 의 최신 비동기 I/O 인터페이스
- 사용자 공간과 커널 공간이 공유하는 ring buffer로 시스템 콜 횟수 최소화
- 디스크 I/O, 네트워크 모두 비동기로 처리 가능
- **고성능 DB / 스토리지 엔진**에서 채택 (RocksDB, ScyllaDB 등)

### 8.4 이벤트 루프
- Node.js, Nginx, Redis 등이 사용하는 모델
- 단일 스레드 + 비동기 I/O로 수만 커넥션 처리
- C10K 문제(만 개 동시 커넥션) 해결의 표준 패턴

## 9. RAID

> **Redundant Array of Independent (Inexpensive) Disks**.
> 여러 디스크를 묶어 **성능 / 내결함성 / 용량** 중 일부를 강화.

### 9.1 주요 RAID 레벨

| 레벨 | 방식 | 디스크 N개 사용 시 용량 | 내결함성 | 성능 | 비고 |
|------|------|---------------------|----------|------|------|
| **RAID 0** | 스트라이핑 | N | **없음** (1개만 죽어도 전체 손실) | 읽/쓰 ↑↑ | 캐시/임시용 |
| **RAID 1** | 미러링 | N/2 | 좋음 (절반까지 가능) | 읽기 ↑, 쓰기 보통 | 단순, 비쌈 |
| **RAID 5** | 스트라이핑 + 분산 패리티 | N-1 | 1개 디스크 손상까지 | 읽기 ↑, 쓰기 페널티 | 일반적 선택 |
| **RAID 6** | 스트라이핑 + 패리티 2개 | N-2 | 2개 동시 손상 가능 | 쓰기 페널티 ↑ | RAID 5의 단점 보완 |
| **RAID 10** | RAID 1 + RAID 0 | N/2 | 우수 | 매우 좋음 | DB에 자주 사용 |

### 9.2 RAID 5의 단점과 RAID 6의 보완
- **RAID 5**: 디스크 1개 손상까지만 견딤. 리빌드 중에 또 다른 디스크 죽으면 데이터 손실
- 대용량 SATA 시대에 **리빌드 시간이 매우 길어짐** → RAID 5만으로 부족
- **RAID 6**: 패리티를 2개로 → 2개까지 손실 견딤

### 9.3 RAID와 백업은 다르다 (중요)
- RAID: **하드웨어 장애에 대비**
- 백업: 실수 삭제, 랜섬웨어, 코드 버그로 인한 데이터 손상에 대비
- RAID가 있어도 **별도 백업 필수** (RAID는 "디스크 1개 죽음"만 해결)

### 9.4 소프트웨어 RAID vs 하드웨어 RAID
- **하드웨어**: 전용 컨트롤러. 성능 좋지만 컨트롤러 고장 시 복구 어려움
- **소프트웨어**: Linux mdadm, ZFS, Btrfs 내장. 유연하고 컨트롤러 의존성 없음
- 현대는 **소프트웨어 RAID + ZFS/Btrfs 같은 FS 통합**이 추세

## 10. 면접 단골 질문

### Q1. DMA가 무엇이고 왜 쓰는가?
- **CPU 개입 없이** 장치 ↔ 메모리 데이터 전송
- 큰 블록 전송 시 인터럽트 폭주 방지, CPU 활용도 ↑

### Q2. 폴링과 인터럽트의 장단점은?
- 폴링: 반응 빠름 but CPU 점유
- 인터럽트: CPU 효율 ↑ but 인터럽트 처리 비용
- 고속 장치는 **NAPI처럼 둘 결합**

### Q3. 블록 장치와 문자 장치의 차이?
- 블록: 고정 크기, 랜덤 접근. 디스크
- 문자: 바이트 스트림, 순차 접근. 키보드/시리얼

### Q4. RAID 0, 1, 5, 6, 10을 비교하라
→ §9.1 표 참조. 핵심: **성능 / 용량 / 내결함성의 트레이드오프**.

### Q5. RAID 5의 단점이 무엇이고 RAID 6은 어떻게 보완하나?
- RAID 5: 1개 손상까지만. 대용량 디스크 시대에 리빌드 중 추가 손상 위험 ↑
- RAID 6: 패리티 2개로 2개 손상 견딤. 쓰기 비용은 더 큼

### Q6. RAID와 백업은 같은 것인가?
- **No.** RAID는 하드웨어 장애 대비. 실수 삭제/소프트웨어 손상은 RAID로 못 막음. 백업은 별도 필요.

### Q7. select / poll / epoll의 차이?
- select: 비트맵, fd 1024 제한, O(N)
- poll: 제한 없음, O(N)
- epoll: 이벤트 등록 모델, O(1), 고성능 서버 표준

### Q8. C10K 문제란?
- 서버 한 대가 동시 1만 커넥션을 처리해야 한다는 도전
- 해결: 스레드 풀 + 비동기 I/O + 이벤트 루프 (epoll, kqueue)
- 현재는 C10M까지 도전 중
