# 프로세스(Process)와 스레드(Thread)

> **한 줄 정리**
> 프로세스는 **자원 할당의 단위**, 스레드는 **CPU 실행의 단위**.
> 프로세스는 메모리 공간을 독립적으로 갖고, 스레드는 같은 프로세스의 메모리를 공유한다.

---

## 1. 왜 이 개념이 필요한가?

### 1.1 초기 컴퓨터의 문제
초기에는 **하나의 프로그램이 끝나야 다음 프로그램이 실행**되는 단일 작업(Single Tasking) 방식이었다.
CPU는 빠른데 입출력(I/O)은 느려서 CPU가 놀고 있는 시간이 너무 많았다.

→ **여러 프로그램을 동시에 실행시키자** → 멀티프로그래밍 → **프로세스 개념 등장**

### 1.2 프로세스만으로는 부족한 이유
프로세스는 독립적이라 안전하지만, 그만큼 무겁다.
- 새 프로세스 생성 시 메모리 공간 전체 복사 (fork 시 Copy-on-Write로 완화되긴 함)
- 프로세스 간 통신(IPC)이 비싸다 (커널 개입 필요)
- 컨텍스트 스위칭 비용이 크다

→ **하나의 프로그램 안에서 여러 작업을 동시에 처리하려면?** → **스레드 개념 등장**

## 2. 프로세스(Process)

### 2.1 정의
> **실행 중인 프로그램**. OS로부터 자원을 할당받는 작업의 단위.

- 프로그램(Program): 디스크에 저장된 정적 코드 (예: `chrome.exe`)
- 프로세스(Process): 메모리에 적재되어 실행 중인 인스턴스

### 2.2 프로세스의 메모리 구조

```
높은 주소 ┌────────────────┐
         │     Stack      │  ← 함수 호출, 지역 변수 (위 → 아래로 자람)
         │       ↓        │
         │                │
         │       ↑        │
         │      Heap      │  ← 동적 할당 (malloc/new) (아래 → 위로 자람)
         ├────────────────┤
         │      BSS       │  ← 초기화되지 않은 전역/정적 변수
         ├────────────────┤
         │      Data      │  ← 초기화된 전역/정적 변수
         ├────────────────┤
         │      Text      │  ← 실행 코드 (read-only)
낮은 주소 └────────────────┘
```

| 영역 | 설명 | 예시 |
|------|------|------|
| **Text (Code)** | 실행 코드. 읽기 전용 | 컴파일된 머신 코드 |
| **Data** | 초기화된 전역/정적 변수 | `static int x = 10;` |
| **BSS** | 초기화되지 않은 전역/정적 변수 | `static int y;` |
| **Heap** | 동적 할당 영역 (런타임에 결정) | `malloc()`, `new` |
| **Stack** | 함수 호출 시 생성. 지역 변수, 매개변수 | 함수의 지역 변수 |

### 2.3 프로세스 제어 블록 (PCB, Process Control Block)
OS가 프로세스를 관리하기 위해 만드는 자료구조. 컨텍스트 스위칭 때 핵심.

**PCB가 담고 있는 정보:**
- 프로세스 식별자 (PID)
- 프로세스 상태 (Ready, Running, Waiting, ...)
- 프로그램 카운터 (PC) — 다음 실행할 명령어 주소
- CPU 레지스터 값
- 메모리 관리 정보 (페이지 테이블, 세그먼트 정보)
- 입출력 상태 정보 (열린 파일 목록 등)
- 계정 정보 (CPU 사용 시간, 우선순위)

### 2.4 프로세스 상태(State)

```
        [New] ──admit──▶ [Ready] ◀──interrupt── [Running]
                            │            │           │
                            └──dispatch──┘           │
                                                     │
                         [Waiting] ◀──I/O or event── ┤
                            │                        │
                            └───I/O 완료─────────────┘
                                                     │
                                                     ▼
                                                [Terminated]
```

| 상태 | 의미 |
|------|------|
| **New** | 프로세스가 생성되는 중 |
| **Ready** | CPU 할당을 기다리는 상태 |
| **Running** | CPU에서 실행 중 |
| **Waiting (Blocked)** | I/O 또는 이벤트를 기다리는 상태 |
| **Terminated** | 실행 종료 |

## 3. 스레드(Thread)

### 3.1 정의
> **프로세스 내에서 실제로 작업을 수행하는 실행 흐름의 단위**.
> 같은 프로세스의 자원(Heap, Data, Code)을 공유한다.

### 3.2 스레드의 메모리 구조

```
┌─────────────────────────────────────┐
│           Process                   │
│  ┌─────┐ ┌─────┐ ┌─────┐            │
│  │Stack│ │Stack│ │Stack│ ← 스레드별 독립
│  │ T1  │ │ T2  │ │ T3  │            │
│  └─────┘ └─────┘ └─────┘            │
│  ┌────────────────────────────────┐ │
│  │      Heap (공유)               │ │
│  ├────────────────────────────────┤ │
│  │      Data / BSS (공유)         │ │
│  ├────────────────────────────────┤ │
│  │      Text (공유)               │ │
│  └────────────────────────────────┘ │
└─────────────────────────────────────┘
```

**스레드마다 독립적으로 갖는 것:**
- Stack (지역 변수, 함수 호출 정보)
- PC (Program Counter)
- 레지스터 값
- 스레드 ID

**스레드끼리 공유하는 것:**
- Heap (동적 할당 영역)
- Data / BSS (전역 변수)
- Text (코드)
- 열린 파일, 시그널 핸들러 등 프로세스 자원

### 3.3 왜 Stack만 분리할까?
- 함수 호출 흐름은 **각 실행 단위마다 독립적**이어야 한다.
- 지역 변수와 호출 스택을 공유하면 멀티스레드가 의미가 없다.
- 반대로 Heap을 공유하기 때문에 **스레드 간 데이터 공유가 쉽다** (그래서 동기화 문제도 생긴다).

## 4. 프로세스 vs 스레드 — 핵심 비교

| 항목 | 프로세스 | 스레드 |
|------|----------|--------|
| **단위** | 자원 할당의 단위 | CPU 실행의 단위 |
| **메모리** | 독립적인 메모리 공간 | 같은 프로세스 메모리 공유 |
| **생성 비용** | 무겁다 (메모리 복사 등) | 가볍다 (Stack만 할당) |
| **컨텍스트 스위칭** | 비싸다 (페이지 테이블 등 교체) | 저렴하다 (같은 주소 공간) |
| **통신 방식** | IPC 필요 (Pipe, Socket, 공유 메모리 등) | 전역 변수, Heap으로 직접 공유 |
| **안정성** | 한 프로세스가 죽어도 다른 프로세스 영향 없음 | 한 스레드가 죽으면 전체 프로세스 영향 |
| **동기화** | 거의 불필요 | 락(Lock), 세마포어 등 필요 |

## 5. 멀티 프로세스 vs 멀티 스레드

### 5.1 멀티 프로세스
**장점**
- 안정성 ↑ (격리)
- 보안 ↑ (메모리 보호)
- 디버깅이 비교적 단순

**단점**
- 메모리 사용량 ↑
- IPC 오버헤드
- 컨텍스트 스위칭 비용 ↑

**언제 쓰나?**
- 안정성이 중요한 시스템 (Chrome 탭마다 프로세스 분리 → 한 탭이 죽어도 다른 탭은 살아남음)
- CPU 바운드 작업 + 격리 필요 (Node.js의 `cluster`, Python의 `multiprocessing` for CPU-bound)
- 보안 샌드박싱 (Docker, 브라우저 렌더러)

### 5.2 멀티 스레드
**장점**
- 자원 효율적 (메모리 절약)
- 컨텍스트 스위칭 빠름
- 데이터 공유가 쉬움
- 응답성 ↑ (UI 스레드 따로, 작업 스레드 따로)

**단점**
- 동기화 이슈 (Race Condition, Deadlock)
- 한 스레드 죽으면 전체 영향
- 디버깅 어려움 (재현 안 되는 버그 多)

**언제 쓰나?**
- I/O 바운드 작업이 많은 서버 (Tomcat, Spring 등)
- 같은 데이터를 여러 작업이 동시에 접근해야 할 때
- UI 응답성 유지가 필요한 GUI 앱

## 6. 컨텍스트 스위칭 (Context Switching)

CPU가 현재 실행 중인 작업의 상태를 저장하고 다른 작업의 상태를 불러와서 실행하는 것.

### 6.1 동작 흐름
1. 현재 실행 중인 프로세스/스레드의 상태(PC, 레지스터 등)를 **PCB(또는 TCB)**에 저장
2. 다음 실행할 프로세스/스레드의 PCB에서 상태를 복원
3. 복원된 PC부터 실행 재개

### 6.2 비용 비교
- **프로세스 컨텍스트 스위칭**: 메모리 주소 공간 교체 → **TLB(Translation Lookaside Buffer) 무효화** → 캐시 미스 발생 → 비쌈
- **스레드 컨텍스트 스위칭**: 같은 프로세스 내 → 메모리 주소 공간 그대로 → 저렴

**그래서 멀티스레드가 멀티프로세스보다 빠르다고 하는 핵심 이유가 여기 있다.**

## 7. 동시성(Concurrency) vs 병렬성(Parallelism)

흔히 헷갈리는 개념. 둘은 다르다.

| 구분 | 동시성 (Concurrency) | 병렬성 (Parallelism) |
|------|---------------------|----------------------|
| **정의** | 여러 작업을 번갈아가며 처리 (논리적) | 여러 작업을 실제로 동시에 처리 (물리적) |
| **CPU 코어** | 1개여도 가능 | 2개 이상 필요 |
| **목적** | 응답성, 효율적 자원 활용 | 처리 속도 향상 |
| **예시** | 싱글코어에서 여러 스레드 빠르게 전환 | 멀티코어에서 스레드 분산 실행 |

## 8. 자주 나오는 면접 포인트

### Q1. 프로세스 간 통신(IPC)에는 무엇이 있나?
- **파이프(Pipe)**: 단방향, 부모-자식 프로세스
- **명명된 파이프(Named Pipe / FIFO)**: 무관한 프로세스 간
- **메시지 큐(Message Queue)**: 커널이 메시지 관리
- **공유 메모리(Shared Memory)**: 가장 빠름. 단, 동기화 필요
- **세마포어(Semaphore)**: 동기화 용도
- **소켓(Socket)**: 네트워크 통신 가능 (다른 머신끼리도)
- **시그널(Signal)**: 비동기 알림

### Q2. 한 프로세스에 스레드가 여러 개일 때, 한 스레드에서 fork()를 호출하면?
- POSIX 기본: **호출한 스레드만 복제됨** (다른 스레드는 자식 프로세스에 없음)
- 이 때문에 자식 프로세스의 뮤텍스가 잠긴 채로 남는 등 위험. 그래서 `fork()` 이후 `exec()` 패턴이 권장됨.

### Q3. 스레드는 무조건 많을수록 좋은가?
**No.**
- 컨텍스트 스위칭 비용 증가
- 동기화 오버헤드 증가
- **CPU 바운드**: 코어 수 ± 1 정도가 적절
- **I/O 바운드**: 더 많이 두어도 됨 (대기 시간이 많아서)
- Amdahl's Law: 직렬 부분이 있으면 병렬화 효과는 제한된다.

### Q4. Chrome이 탭마다 프로세스를 분리하는 이유는?
- **안정성**: 한 탭 크래시가 전체 브라우저에 영향을 안 줌
- **보안**: 사이트별 메모리 격리 (Site Isolation) → Spectre/Meltdown 같은 사이드 채널 공격 방어
- **자원 회수 용이**: 탭 닫을 때 프로세스 단위로 메모리 회수

### Q5. PCB와 TCB의 차이?
- **PCB (Process Control Block)**: 프로세스 전체 정보 (메모리 맵, 열린 파일, 시그널 핸들러 등 포함)
- **TCB (Thread Control Block)**: 스레드 고유 정보 (Stack 포인터, PC, 레지스터, 스레드 상태)
- TCB는 PCB가 가리키는 자원을 공유하며, 스레드별로 별도 존재

## 9. 실무 연결 — 어디서 마주치는가?

- **Spring Boot / Tomcat**: 요청마다 스레드 풀에서 스레드 할당 (멀티스레드 서버)
- **Node.js**: 싱글 스레드 이벤트 루프 + libuv의 워커 스레드 풀 (I/O 작업용)
- **Python**: GIL(Global Interpreter Lock) 때문에 CPU 바운드는 멀티프로세싱이 사실상 표준
- **Chrome**: 멀티 프로세스 아키텍처 (Browser / Renderer / GPU / Network 프로세스 분리)
- **Docker**: 컨테이너는 호스트의 프로세스. cgroups + namespace로 격리된 프로세스
