# 파일 시스템 (File System)

> **한 줄 정리**
> 디스크라는 **블록 단위 저장소를 "이름 있는 파일"이라는 추상화**로 만드는 OS의 한 계층.
> 핵심은 **메타데이터(inode 등)**, **디렉토리 구조**, **할당 방식**, **신뢰성(저널링)**.

---

## 1. 왜 파일 시스템이 필요한가?

### 1.1 원시 디스크의 문제
- 디스크는 **블록(보통 4KB)의 거대한 배열**일 뿐
- "이 데이터는 누구 거?", "어디서 시작해서 어디서 끝나?", "이름이 뭐야?" — 아무것도 모름
- 여러 프로그램이 동시에 쓰면 충돌, 손상

### 1.2 파일 시스템의 역할
- **이름 → 데이터 매핑** (사람이 쓰기 쉬운 추상화)
- **공간 관리** (할당/회수, 단편화 관리)
- **계층 구조** (디렉토리 트리)
- **보호** (권한, 소유권)
- **신뢰성** (장애 시 일관성 유지)
- **공유 & 동시성** (락, 공유 파일)

---

## 2. 핵심 개념

### 2.1 파일 (File)
- 이름과 메타데이터를 가진 데이터의 논리적 단위
- 메타데이터: 크기, 소유자, 권한, 생성/수정 시각, 위치 정보 등
- 데이터: 실제 내용 (바이트 시퀀스)

### 2.2 디렉토리 (Directory)
- 이름 → 파일 매핑 테이블 (디렉토리도 일종의 파일)
- 단일 레벨 → 트리 → DAG → 그래프 (심볼릭 링크)

### 2.3 마운트 (Mount)
- 별도 파일 시스템을 디렉토리 트리에 붙이기
- 예: USB를 `/mnt/usb`에 마운트하면 그 아래로 다른 FS 접근
- Linux의 `/proc`, `/sys`처럼 가상 파일 시스템도 마운트됨

### 2.4 경로 (Path)
- **절대 경로**: `/home/user/file.txt` (루트부터)
- **상대 경로**: `./file.txt`, `../sibling/file.txt` (현재 위치 기준)

## 3. 파일 시스템 구조 (디스크 레이아웃)

전형적인 Unix 계열 파일 시스템:

```
┌──────────┬──────────┬──────────┬─────────┬─────────────┐
│ Boot     │Super     │  Inode   │  Data   │             │
│ Block    │Block     │  Table   │  Blocks │     ...     │
└──────────┴──────────┴──────────┴─────────┴─────────────┘
```

| 영역 | 역할 |
|------|------|
| **Boot Block** | 부팅 코드 (없는 FS도 있음) |
| **Superblock** | 파일 시스템 전체 메타데이터 (크기, inode 수, free 블록 비트맵 위치 등) |
| **Inode Table** | 파일별 메타데이터 모음 |
| **Data Blocks** | 실제 파일 내용 |
| **Free Block / Inode Bitmap** | 빈 공간 추적 |

## 4. inode (Index Node)

> **파일의 메타데이터 + 데이터 블록 위치를 담은 자료구조**. 파일 이름은 **inode에 없다**.

### 4.1 inode가 담는 것
- 파일 타입 (정규 파일 / 디렉토리 / 심볼릭 링크 / 디바이스)
- 권한 (rwx for owner/group/other)
- 소유자 UID, GID
- 크기
- 타임스탬프: atime(접근), mtime(수정), ctime(메타데이터 변경)
- **링크 수 (link count)**
- **데이터 블록 포인터들** (가장 중요)

### 4.2 데이터 블록 포인터 — 다단계 인덱싱
inode에 모든 블록 주소를 직접 담으면 파일 크기 제한 ↑ + 작은 파일에도 낭비.
**작은 파일은 빠르게, 큰 파일도 가능**하도록 다단계 인덱싱 사용.

```
inode
├── Direct Pointer × 12     → 데이터 블록을 직접 가리킴
├── Single Indirect          → 포인터 블록 → 데이터 블록
├── Double Indirect          → 포인터 블록 → 포인터 블록 → 데이터 블록
└── Triple Indirect          → 3단계
```

**예시 (4KB 블록, 4byte 포인터)**
- Direct: 12 × 4KB = 48KB
- Single Indirect: 1024 × 4KB = 4MB
- Double Indirect: 1024² × 4KB = 4GB
- Triple Indirect: 1024³ × 4KB = 4TB

→ 작은 파일은 inode만 보면 끝 (빠름), 큰 파일도 표현 가능.

### 4.3 파일 이름은 어디에?
- **디렉토리 엔트리(dentry)에 있음** (이름 ↔ inode 번호)
- 그래서 같은 inode를 여러 이름으로 가리키는 것이 **하드 링크**

## 5. 디렉토리

### 5.1 디렉토리의 본질
- 디렉토리 = 특별한 파일 (이름 → inode 번호 매핑 테이블)
- 디렉토리에 entry 추가/삭제 = 파일 추가/삭제

### 5.2 디렉토리 구현
- **리스트**: 작고 단순 (수십 개 파일까지)
- **해시 테이블**: 빠른 검색 (수천 개 파일)
- **B-Tree / B+Tree**: ext4의 HTree, NTFS의 B-Tree (대규모)

### 5.3 하드 링크 vs 심볼릭 링크

| 종류 | 동작 | 특징 |
|------|------|------|
| **Hard Link** | 같은 inode를 다른 이름으로 가리킴 | 원본 삭제해도 살아있음 (링크 수 0이 되어야 진짜 삭제), 같은 FS 내에서만, 디렉토리 불가 |
| **Symbolic Link (소프트 링크)** | 다른 경로를 가리키는 작은 파일 | 원본 삭제 시 깨짐, 다른 FS 가능, 디렉토리 가능 |

```
# 하드 링크
ln file.txt hard.txt          # file.txt와 hard.txt는 같은 inode

# 심볼릭 링크
ln -s file.txt soft.txt       # soft.txt는 "file.txt"라는 경로를 저장
```

## 6. 파일 할당 방식 (Allocation)

데이터 블록을 어떻게 모아둘지.

### 6.1 연속 할당 (Contiguous)
- 파일을 연속된 블록에 저장
- **장점**: 순차 접근 + 랜덤 접근 모두 빠름 (단순 계산)
- **단점**: 외부 단편화, 파일 크기 변경 어려움
- **사용**: 옛 시스템, 일부 임베디드

### 6.2 연결 할당 (Linked)
- 각 블록에 다음 블록 포인터
- **장점**: 단편화 없음, 크기 변경 자유
- **단점**: 랜덤 접근 느림 (포인터 따라가야), 포인터 손상 시 데이터 손실
- **변형**: **FAT (File Allocation Table)** — 포인터를 따로 모아 메모리에 캐싱

### 6.3 인덱스 할당 (Indexed)
- inode 같은 인덱스 블록이 데이터 블록 포인터 보관
- **장점**: 랜덤 접근 빠름, 단편화 없음
- **단점**: 인덱스 블록 오버헤드 (작은 파일에 비용)
- **사용**: Unix 계열 (ext2/3/4, UFS)

### 6.4 익스텐트 (Extent)
- 연속된 블록 범위를 (시작, 길이)로 표현
- 큰 파일에 효율적 (포인터 폭증 방지)
- **사용**: ext4, NTFS, XFS, APFS

## 7. 빈 공간 관리

### 7.1 비트맵 (Bitmap)
- 블록마다 1비트 (0=free, 1=used)
- 검색 빠름, 연속 블록 찾기 쉬움
- 작은 메모리 (1TB / 4KB = 32MB 비트맵)

### 7.2 Free List
- 빈 블록들을 연결 리스트로
- 메모리 효율 ↑ but 검색 ↓, 연속 블록 찾기 어려움

대부분 현대 FS는 **비트맵 + 알로케이션 그룹(Allocation Group)** 조합 사용.

## 8. 신뢰성 — 저널링 (Journaling)

### 8.1 문제
디스크 쓰기 도중 전원이 나가면?
- 파일 데이터는 썼는데 메타데이터 갱신 전 → 데이터는 있지만 가리켜지지 않음 (orphan)
- 메타데이터만 갱신되고 데이터 안 씀 → 가비지를 가리킴
- **파일 시스템 일관성 깨짐**

복구는 `fsck`로 가능하지만 TB 규모면 **수십 분~수 시간** 걸림.

### 8.2 저널링의 아이디어
- 변경 사항을 **저널(로그)에 먼저 기록 (Commit)**
- 실제 메타데이터/데이터에 반영 (Checkpoint)
- 크래시 시: 저널을 보고 미완료 작업을 **재시도(Redo)** 또는 **무시(Undo)**

### 8.3 저널링 수준 (ext3/4 기준)

| 모드 | 저널에 기록 | 안정성 | 속도 |
|------|------------|--------|------|
| **journal** | 메타데이터 + 데이터 | 가장 안전 | 느림 (모든 게 2번 쓰임) |
| **ordered** | 메타데이터만. 단, 데이터 먼저 디스크에 쓰고 → 메타데이터 저널 | 균형 (기본값) | 중간 |
| **writeback** | 메타데이터만, 순서 보장 X | 빠름 | 일관성 약함 |

### 8.4 COW (Copy-on-Write) 파일 시스템
- 저널 없이 **새 위치에 쓰고 포인터를 원자적으로 교체**
- 항상 일관 상태 유지
- 스냅샷, 체크섬, 복제 등 강력한 기능
- **예**: ZFS, Btrfs, APFS

## 9. 가상 파일 시스템 (VFS, Virtual File System)

> 서로 다른 파일 시스템들을 **하나의 API로 추상화**하는 커널 계층.

```
┌──────────────────────────┐
│   User: open/read/write   │
└────────────┬──────────────┘
             ↓
┌──────────────────────────┐
│           VFS             │   ← 공통 인터페이스
└──┬──────┬──────┬──────┬──┘
   ↓      ↓      ↓      ↓
 ext4   xfs   ntfs   nfs(네트워크)
```

덕분에:
- 같은 `read()` 코드가 ext4든 NFS든 동작
- `/proc`, `/sys` 같은 **가상 파일 시스템**도 같은 인터페이스로 노출

## 10. 주요 파일 시스템 비교

| 파일 시스템 | 등장 | 특징 |
|------------|------|------|
| **FAT32** | 1996 | 단순, 호환성 ↑. 단일 파일 4GB 제한. USB/SD에 많이 씀 |
| **NTFS** | 1993 | Windows 기본. 저널링, ACL, 압축, 암호화 |
| **ext2** | 1993 | Linux 초기. 저널 없음 |
| **ext3** | 2001 | ext2 + 저널링 |
| **ext4** | 2008 | 익스텐트, 더 큰 파일 지원, 백워드 호환 |
| **XFS** | 1993 (SGI) | 대용량/고성능. RHEL 기본 |
| **Btrfs** | 2009 | COW, 스냅샷, RAID 내장 |
| **ZFS** | 2005 (Sun) | COW, 무결성 체크섬, RAID-Z, 거대한 풀 |
| **APFS** | 2017 (Apple) | COW, 스냅샷, SSD 최적화. macOS/iOS |
| **HFS+** | 1998 (Apple) | APFS 이전 macOS 기본 |

## 11. 캐싱 — 페이지 캐시 / 버퍼 캐시

### 11.1 페이지 캐시 (Page Cache)
- 디스크 I/O는 메모리보다 **수십만 배 느림**
- 한 번 읽은 데이터를 RAM에 캐싱 → 다음 접근은 빠름
- Linux는 **남는 메모리를 모두 페이지 캐시로** 사용 (`free` 명령의 "buff/cache")

### 11.2 쓰기 정책
- **Write-Through**: 즉시 디스크에 기록. 안전하지만 느림
- **Write-Back**: 메모리에만 기록 후 나중에 flush. 빠르지만 크래시 시 손실 가능
- Linux 기본은 write-back + 주기적 flush + `fsync()`로 명시적 동기화

### 11.3 fsync()의 중요성
- DB, 로그 시스템에서 트랜잭션 커밋 시점에 반드시 호출
- "쓴 것 같은데 사실은 메모리에만 있어서 크래시 시 사라짐" 사고를 방지

## 12. 파일 열기 메커니즘

### 12.1 파일 디스크립터 (File Descriptor)
- 프로세스가 가진 작은 정수 (0=stdin, 1=stdout, 2=stderr)
- 커널 내부의 file descriptor table → file table → inode로 연결

```
프로세스 fd table          시스템 file table         inode table
┌────┐                    ┌────────────────┐       ┌─────────┐
│ 3  │ ───────────────→  │ offset, flags, │ ────→ │ inode A │
│ 4  │ ───────────────→  │ ref count, ... │       └─────────┘
└────┘                    └────────────────┘
```

### 12.2 같은 파일을 두 번 열면?
- 각각 별도의 file table entry (각자 offset 보유)
- 가리키는 inode는 같음
- 그래서 두 fd가 같은 파일을 서로 다른 위치에서 읽을 수 있음

### 12.3 dup / fork 와의 관계
- `dup` / `dup2`: 같은 file table entry를 가리키는 fd 복제 (offset 공유)
- `fork()` 후 자식: 같은 file table entry 공유 → 부모/자식이 같은 offset

## 13. 면접 단골 질문

### Q1. inode란 무엇인가? 이름은 inode에 있는가?
- inode는 파일의 메타데이터 + 블록 위치 정보를 담은 구조
- **이름은 inode에 없고, 디렉토리 엔트리에 있음**
- 하드 링크가 가능한 이유

### Q2. 하드 링크와 심볼릭 링크의 차이?
→ §5.3 표 참조

### Q3. 저널링은 왜 필요한가? `journal` / `ordered` / `writeback`의 차이는?
- 크래시 시 일관성 보장 + 빠른 복구
- 모드는 안정성과 속도의 트레이드오프 (§8.3)

### Q4. 디렉토리도 파일인가?
- **Yes.** 단지 내용이 "이름 → inode 번호" 매핑인 특별한 파일

### Q5. 큰 파일은 inode에 어떻게 저장되나?
- 다단계 인덱싱 (직접/단일/이중/삼중 간접 포인터) — §4.2
- 현대 FS는 익스텐트 사용

### Q6. fsync()를 왜 호출해야 하나?
- 페이지 캐시(Write-Back)에 머무는 데이터를 디스크로 강제 flush
- DB의 트랜잭션 durability(D), 로그 안정성에 필수
- fsync 없이 write만 했다면 크래시 시 데이터가 사라질 수 있음

### Q7. FAT와 ext4의 차이?
- FAT: 단순한 링크드 리스트 기반, 메타데이터 빈약, 권한 없음
- ext4: inode + 익스텐트 + 저널링 + 권한/ACL + 대용량 지원

### Q8. VFS가 왜 필요한가?
- 여러 파일 시스템을 통일된 API로 다루기 위함
- 새 FS 추가 시 유저 코드 변경 불필요
- `/proc`, `/sys` 같은 가상 파일 시스템도 같은 인터페이스 제공
