# 메모리 관리 (Memory Management)

> **한 줄 정리**
> 한정된 물리 메모리를 여러 프로세스가 효율적으로 나눠 쓰도록 **할당 / 보호 / 추상화**하는 일.
> 핵심 아이디어는 **가상 주소 ↔ 물리 주소 매핑**과 **단편화 최소화**.

---

## 1. 왜 메모리 관리가 필요한가?

### 1.1 해결해야 할 문제들
- **다중 프로세스**: 여러 프로세스가 같은 RAM을 공유해야 함
- **보호**: 한 프로세스가 다른 프로세스 메모리를 침범하면 안 됨
- **공간 부족**: 물리 메모리보다 더 큰 작업을 어떻게 돌릴 것인가
- **단편화**: 메모리 곳곳에 빈 조각이 생기면 큰 블록 할당 실패
- **공유**: 같은 라이브러리(libc 등)는 한 번만 적재해서 여러 프로세스가 공유

### 1.2 핵심 아이디어: 가상 메모리(Virtual Memory)
- 프로세스는 **자기만의 연속된 가상 주소 공간**을 본다고 착각
- OS + MMU(Memory Management Unit)가 **가상 → 물리 주소 변환**
- 덕분에 보호, 격리, 공유, 스왑이 가능

---

## 2. 메모리 계층 (Memory Hierarchy)

빠르고 작은 것 → 느리고 큰 것 순서.

```
        ↑ 빠름, 비쌈, 작음
   ┌───────────┐
   │ Register  │   1 cycle
   ├───────────┤
   │  L1 Cache │   ~4 cycles
   ├───────────┤
   │  L2 Cache │   ~10 cycles
   ├───────────┤
   │  L3 Cache │   ~40 cycles
   ├───────────┤
   │   RAM     │   ~100 cycles
   ├───────────┤
   │  SSD/HDD  │   수십만 cycles
   ├───────────┤
   │  Network  │
   └───────────┘
        ↓ 느림, 쌈, 큼
```

**지역성(Locality)** 덕분에 캐시가 효과 있음:
- **시간 지역성 (Temporal)**: 최근 접근한 것은 또 접근될 가능성↑
- **공간 지역성 (Spatial)**: 접근한 주소 근처도 곧 접근될 가능성↑

## 3. 주소 바인딩 (Address Binding)

가상/논리 주소 → 물리 주소 결정 시점:

| 단계 | 시점 | 비고 |
|------|------|------|
| **컴파일 타임 바인딩** | 컴파일 시 절대 주소 결정 | 위치 바뀌면 재컴파일 필요. 임베디드 외엔 안 씀 |
| **로드 타임 바인딩** | 프로그램 적재 시 결정 | 적재 후 위치 고정 |
| **실행 타임 바인딩** | 실행 중에 결정 (가장 일반적) | MMU가 매번 변환. **현대 OS의 기본** |

## 4. 연속 할당 (Contiguous Allocation)

가상 메모리 이전의 단순한 방식. 프로세스를 **메모리에 통째로 연속해서** 적재.

### 4.1 고정 분할 (Fixed Partition)
- 메모리를 고정 크기 파티션으로 분할
- 단순하지만 **내부 단편화** 발생 (할당 받은 영역 안에서 남는 공간)

### 4.2 가변 분할 (Variable Partition)
- 프로세스 크기에 맞춰 가변 할당
- **외부 단편화** 발생 (총 가용 공간은 충분한데 연속된 큰 블록이 없음)

### 4.3 할당 전략
빈 공간(hole)이 여러 개일 때 어디에 넣을지:

| 전략 | 방식 | 특징 |
|------|------|------|
| **First Fit (최초적합)** | 처음 만나는 충분한 hole | 빠름, 메모리 앞부분에 단편화 집중 |
| **Best Fit (최적적합)** | 가장 작은 충분한 hole | 큰 hole 보존, 작은 자투리 多 |
| **Worst Fit (최악적합)** | 가장 큰 hole | 자투리도 큼 → 재사용 가능성, 큰 hole 소진↓ |

일반적으로 **First Fit과 Best Fit이 Worst Fit보다 우수**. First Fit이 가장 빠름.

### 4.4 단편화 (Fragmentation)
| 종류 | 발생 위치 | 설명 |
|------|----------|------|
| **내부 단편화** | 할당된 영역 **안** | 요청한 것보다 많이 할당되어 남는 공간 |
| **외부 단편화** | 할당 영역 **사이** | 빈 조각들이 흩어져서 큰 연속 블록 부족 |

**해결**:
- **컴팩션(Compaction)**: 빈 공간을 한쪽으로 몰기 — 비용 큼
- **페이징/세그멘테이션** — 애초에 연속 할당을 포기

## 5. 페이징 (Paging)

> **물리 메모리를 고정 크기 프레임(Frame), 가상 메모리를 고정 크기 페이지(Page)로 나누고 자유롭게 매핑.**
> 외부 단편화를 근본적으로 없앤다.

### 5.1 기본 구조
- **페이지(Page)**: 가상 메모리의 고정 크기 블록 (보통 4KB)
- **프레임(Frame)**: 물리 메모리의 같은 크기 블록
- **페이지 테이블(Page Table)**: 각 프로세스마다 존재, "페이지 → 프레임" 매핑

### 5.2 주소 변환

```
가상 주소 = [페이지 번호 | 페이지 오프셋]
                ↓
         [페이지 테이블 룩업]
                ↓
        프레임 번호
                ↓
물리 주소 = [프레임 번호 | 페이지 오프셋]
```

**예시**: 32비트 주소, 4KB 페이지
- 페이지 오프셋: 하위 12비트 (2¹² = 4096)
- 페이지 번호: 상위 20비트 (2²⁰ = 1M 페이지)

### 5.3 페이지 테이블 문제와 해결

**문제**: 32비트 + 4KB 페이지 → 페이지 테이블 엔트리 100만 개. 프로세스마다 4MB 필요. 64비트면 천문학적.

**해결책들**:

#### (1) 다단계 페이지 테이블 (Multi-level Page Table)
- 페이지 테이블 자체를 페이징
- 사용하지 않는 영역은 메모리에 안 올림
- x86-64는 보통 **4단계** (PML4 → PDPT → PD → PT)

#### (2) 해시 페이지 테이블 (Hashed Page Table)
- 가상 페이지 번호를 해시 → 충돌 시 체이닝
- 64비트 시스템에서 유리

#### (3) 역 페이지 테이블 (Inverted Page Table)
- 프레임당 한 엔트리만 (전체 시스템에 하나)
- 메모리는 절약되지만 검색 비용 ↑

### 5.4 TLB (Translation Lookaside Buffer)
> 페이지 테이블 룩업을 **캐시**하는 MMU 내부의 작은 고속 메모리.

**없으면?** 메모리 접근 1번에 페이지 테이블 룩업 N번(다단계라면) + 실제 데이터 접근 = 매우 느림.

**TLB 동작**
- TLB Hit: 즉시 변환 (1 cycle)
- TLB Miss: 페이지 테이블 워크 (수십 cycle) → TLB에 채움

**TLB 무효화 (TLB Flush)**
- 컨텍스트 스위칭 시 프로세스마다 페이지 테이블이 다르므로 TLB를 비워야 함 → 비싼 이유
- **ASID (Address Space ID)**: 각 엔트리에 프로세스 ID를 붙여서 무효화 회피

### 5.5 페이지 테이블 엔트리(PTE)의 정보
- 프레임 번호
- **Valid/Invalid bit**: 페이지가 메모리에 있는가
- **Protection bits**: 읽기/쓰기/실행 권한
- **Dirty bit**: 수정되었는가 (디스크에 다시 써야 하나)
- **Reference bit**: 최근 접근되었는가 (페이지 교체 알고리즘용)
- **Cache disable**: 메모리 매핑 I/O 영역 등

### 5.6 페이징의 장단점
**장점**: 외부 단편화 없음, 메모리 공유 쉬움(같은 프레임 매핑)
**단점**: 내부 단편화 (페이지 마지막 부분), 페이지 테이블 메모리 오버헤드, TLB 미스 비용

## 6. 세그멘테이션 (Segmentation)

> 메모리를 **논리적 단위**(코드/데이터/스택/힙 등)로 나누어 **가변 크기**로 할당.

### 6.1 구조
- 세그먼트 테이블: 각 세그먼트의 (base, limit) 저장
- 가상 주소 = [세그먼트 번호 | 오프셋]
- 변환: `base + offset` (단, `offset < limit` 확인)

### 6.2 페이징 vs 세그멘테이션

| 항목 | 페이징 | 세그멘테이션 |
|------|--------|-------------|
| **블록 크기** | 고정 | 가변 (논리 단위) |
| **단편화** | 내부 | 외부 |
| **공유** | 가능 (페이지 단위) | 가능 (세그먼트 단위, 더 자연스러움) |
| **보호** | 페이지 단위 | 세그먼트 단위 (논리 단위라 더 자연스러움) |
| **사용** | 거의 모든 현대 OS | 단독 사용은 드묾 |

### 6.3 세그멘테이션 + 페이징 (Segmented Paging)
- 세그먼트 안을 다시 페이징 → 두 방식의 장점 결합
- 과거 Intel x86이 사용 (현재는 x86-64에서 세그멘테이션이 거의 무효화됨)

## 7. 가상 메모리 & 디맨드 페이징

### 7.1 디맨드 페이징 (Demand Paging)
> **실제로 접근할 때만 페이지를 메모리에 적재**. 게으른 로딩.

- 프로세스 시작 시 모든 페이지를 안 올림
- 페이지 테이블의 Valid bit = 0 → 접근 시 **페이지 폴트(Page Fault)** 발생
- OS가 디스크에서 페이지를 가져와 프레임에 적재 → 다시 실행

### 7.2 페이지 폴트 처리 흐름
1. CPU가 페이지에 접근 → Valid bit = 0
2. 트랩 발생 → OS의 페이지 폴트 핸들러
3. 디스크에서 페이지 로드 (수 ms — CPU 입장에서는 영겁)
4. 빈 프레임이 없으면 **페이지 교체 알고리즘**으로 희생자 선택
5. PTE 갱신 → 명령어 재실행

**페이지 폴트 비용**: 메모리 접근의 약 **10⁶배** (마이크로초 vs 나노초 단위).
→ 페이지 폴트율이 1%만 되어도 실효 메모리 접근 시간은 100배 느려짐.

### 7.3 스왑(Swap)
- 메모리 부족 시 일부 페이지를 디스크(스왑 영역)로 이동
- 다시 필요해지면 **페이지 폴트 → 디스크 → 메모리 로드**
- 너무 자주 일어나면 **쓰래싱(Thrashing)**

## 8. 페이지 교체 알고리즘 (Page Replacement)

빈 프레임이 없을 때 어떤 페이지를 디스크로 보낼지.

### 8.1 OPT (Optimal)
- **앞으로 가장 오래 쓰지 않을 페이지** 교체
- 미래를 알 수 없으니 **이론적 최적치**로만 사용
- 다른 알고리즘 평가 기준

### 8.2 FIFO (First-In, First-Out)
- 가장 먼저 들어온 페이지 교체
- 구현 단순
- **Belady's Anomaly**: 프레임 수를 늘렸는데 페이지 폴트가 더 늘어나는 역설

### 8.3 LRU (Least Recently Used)
- **가장 오랫동안 사용되지 않은** 페이지 교체
- 시간 지역성을 잘 활용 → 성능 좋음
- 구현 비용: 매 접근마다 타임스탬프 갱신 또는 더블 링크드 리스트 유지 → 비쌈

### 8.4 LRU 근사 알고리즘
**Reference Bit (Second Chance / Clock Algorithm)**
- 각 페이지에 1비트 Reference bit
- 접근하면 1로 세팅, 교체 후보면 0인 페이지를 찾음
- 1인 페이지는 0으로 바꾸고 패스 (한 번의 기회)
- **원형 큐(Clock)**로 순회 → 효율적

**NRU (Not Recently Used)**: Reference bit + Dirty bit 조합으로 4단계 분류

### 8.5 LFU (Least Frequently Used)
- 가장 적게 사용된 페이지 교체
- 초기에 자주 쓰였다가 지금은 안 쓰이는 페이지가 살아남는 단점

### 8.6 비교

| 알고리즘 | 폴트율 | 구현 비용 | 비고 |
|---------|--------|----------|------|
| OPT | 최저 (이론) | 불가능 | 비교 기준 |
| FIFO | 높음 | 매우 단순 | Belady 이상 |
| LRU | 낮음 | 비쌈 | 일반적으로 우수 |
| Clock | LRU와 유사 | 저렴 | **실무에서 자주 사용** |
| LFU | 변동 | 중간 | 빈도 편향 |

## 9. 쓰래싱 (Thrashing)

> 페이지 폴트가 너무 자주 발생해서 **실제 계산보다 페이지 적재에 시간을 다 쓰는 상태**.

```
CPU 이용률
  │           ╱─────╲
  │          ╱       ╲
  │         ╱         ╲___       ← 쓰래싱 시작
  │        ╱              ╲___
  │_______╱                    ╲__
  └──────────────────────────────→ 프로세스 수
```

프로세스를 늘리면 처음엔 CPU 이용률이 올라가다가, **너무 많아지면 각 프로세스에 할당된 프레임이 부족해져** 페이지 폴트가 폭증.

### 9.1 해결
**Working Set 모델 (Denning)**
- 프로세스가 일정 시간 동안 참조한 페이지 집합
- 워킹셋이 메모리에 모두 있으면 폴트 적음
- 워킹셋 크기 합이 메모리를 넘으면 일부 프로세스를 스왑 아웃

**Page Fault Frequency (PFF)**
- 페이지 폴트율을 측정해서 상한/하한 유지
- 상한 초과 → 프레임 더 할당
- 하한 미만 → 프레임 회수

## 10. 메모리 할당 (User Level)

### 10.1 Heap 할당자 (malloc/free)
- C의 `malloc`, C++의 `new`, JVM의 객체 할당 등
- `brk`/`sbrk` 또는 `mmap` 시스템 콜로 OS에서 큰 청크 받아 관리
- 내부적으로 free list, size class 등 자료구조 사용

### 10.2 단편화 (Heap)
- **외부 단편화**: 작은 자투리가 흩어져 큰 할당 실패 (malloc도 발생)
- **내부 단편화**: 8/16/32바이트 같은 size class로 인해 요청 이상 할당
- **해결**: 메모리 풀, 슬랩 할당자(slab), 가비지 컬렉터의 compaction

### 10.3 가비지 컬렉션 (Garbage Collection)
- JVM, Go, Python(참조 카운팅 + 사이클 GC) 등
- **Mark & Sweep**: 도달 가능 객체 마킹 → 나머지 회수
- **Generational GC**: Young/Old 세대 분리 (대부분 객체는 빨리 죽는다는 가정)
- **Compaction**: 살아있는 객체를 한쪽으로 몰아 단편화 제거

## 11. 메모리 보호 & 공유

### 11.1 보호
- 각 페이지에 권한 비트 (read/write/execute)
- 위반 시 SIGSEGV (segmentation fault)
- **DEP (Data Execution Prevention) / NX bit**: 데이터 영역에서 코드 실행 금지

### 11.2 공유
- 같은 프레임을 여러 페이지 테이블에서 매핑
- 예: `libc`는 한 번만 로드해서 모든 프로세스가 공유
- **Copy-on-Write (CoW)**: `fork()` 시 페이지를 공유로 표시하고, 쓰기 시점에만 복사
  - 덕분에 `fork()`가 빠르고 메모리 효율적

### 11.3 메모리 매핑 파일 (mmap)
- 파일을 가상 주소 공간에 매핑
- 파일 I/O를 메모리 접근처럼 사용 가능
- 페이지 단위로 게으르게 로드 → 디맨드 페이징과 자연스럽게 결합

## 12. 면접 단골 질문

### Q1. 내부 단편화와 외부 단편화의 차이는?
- **내부**: 할당된 블록 **안**에서 낭비 (요청보다 큰 단위로 할당)
- **외부**: 할당 블록 **사이**의 빈 공간이 흩어져 큰 요청 실패
- 페이징 → 외부 단편화 X, 내부 단편화 O (페이지 끝 부분)

### Q2. 페이지 크기를 크게 / 작게 하면?
**크게**:
- 페이지 테이블 작아짐, TLB 적중률 ↑
- 내부 단편화 ↑, 페이지 폴트 시 더 많은 데이터 로드

**작게**:
- 내부 단편화 ↓, 디스크 I/O 효율 ↓
- 페이지 테이블 커짐, TLB 미스 ↑

→ 워크로드에 따라 다름. 일반적으로 4KB가 표준. 대용량 메모리용 **Huge Page (2MB, 1GB)**도 지원.

### Q3. TLB가 왜 중요한가?
- 페이지 테이블 룩업이 매 메모리 접근마다 필요한데, 다단계면 수십 cycle
- TLB Hit면 1 cycle → 메모리 접근 비용을 거의 0으로 만듦
- 컨텍스트 스위칭 시 TLB 무효화가 비싼 이유 = 그만큼 TLB 효과가 크다는 뜻

### Q4. Demand Paging의 장점은?
- 시작이 빠름 (필요한 페이지만 로드)
- 메모리 절약 (안 쓰는 페이지는 안 올림)
- 큰 프로그램도 작은 메모리에서 실행 가능

### Q5. Belady's Anomaly란?
- FIFO에서 **프레임 수를 늘렸는데 페이지 폴트가 더 많아지는 현상**
- LRU 등 stack 알고리즘에서는 발생하지 않음

### Q6. Copy-on-Write가 왜 좋은가?
- `fork()` 직후 자식이 거의 같은 메모리를 쓰는데 즉시 복사하면 낭비
- 페이지를 공유로 표시하고 쓰기가 발생할 때만 복사 → 시간/공간 효율
- 자식이 곧 `exec()`하면 결국 거의 복사 안 일어남

### Q7. 쓰래싱은 왜 발생하고 어떻게 해결하나?
- 프로세스당 프레임이 부족해 페이지 폴트 폭증 → CPU 이용률↓ → 스케줄러가 프로세스 더 추가 → 악순환
- 해결: Working Set 모델, PFF 모니터링, 일부 프로세스 스왑 아웃

### Q8. 가상 메모리 덕분에 가능한 것들은?
- **격리**: 프로세스끼리 주소 공간 분리
- **보호**: 페이지 권한 비트로 침범 차단
- **공유**: 같은 프레임을 여러 페이지에 매핑
- **큰 주소 공간**: 물리보다 큰 프로그램 실행 (스왑)
- **mmap / Copy-on-Write 등 유연한 메모리 모델**
