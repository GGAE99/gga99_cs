# 인터럽트(Interrupt)와 시스템 콜(System Call)

> **한 줄 정리**
> 인터럽트는 CPU에 **비동기 이벤트를 통지**하는 메커니즘이고, 시스템 콜은 **유저가 커널 기능을 호출**하는 통로다.
> 둘 다 결국 **CPU 모드를 사용자 → 커널로 전환**시키는 동일한 하드웨어 메커니즘 위에서 동작한다.

---

## 1. 왜 인터럽트가 필요한가?

### 1.1 폴링(Polling)의 문제
인터럽트 없이 외부 장치 상태를 확인하려면 CPU가 계속 묻는 수밖에 없다.
```
while (true) {
    if (keyboard_has_data()) read_keyboard();
    if (disk_done()) read_disk();
    // ... CPU 시간 다 씀
}
```
→ CPU가 **놀지도, 다른 일도 못 함**. 비효율적.

### 1.2 인터럽트의 아이디어
> 장치가 준비되면 **CPU에 신호를 보내고**, CPU는 평소엔 다른 일을 하다가 **신호가 오면 잠깐 멈추고** 처리.

→ 비동기 이벤트 모델. CPU 활용도 ↑, 응답성 ↑.

---

## 2. 인터럽트의 종류

### 2.1 큰 분류

| 구분 | 발생 원인 | 동기/비동기 | 예시 |
|------|----------|-------------|------|
| **하드웨어 인터럽트 (External / IRQ)** | CPU 외부 장치 | **비동기** | 키보드 입력, 디스크 완료, 타이머 |
| **소프트웨어 인터럽트 (Trap / Exception)** | CPU 내부, 명령어 실행 중 | **동기** | 시스템 콜, 0으로 나누기, 페이지 폴트 |

### 2.2 소프트웨어 인터럽트 세분화

| 종류 | 의미 | 처리 후 |
|------|------|---------|
| **Fault** | 복구 가능한 오류 | 명령어 재실행 (예: 페이지 폴트) |
| **Trap** | 의도적 발생 | 다음 명령어로 진행 (예: 시스템 콜, 디버거 breakpoint) |
| **Abort** | 복구 불가능 | 프로세스 종료 (예: 머신 체크) |

### 2.3 동기 vs 비동기

| 구분 | 의미 |
|------|------|
| **동기 (Synchronous)** | 특정 명령어 실행 때문에 발생 → **재실행 가능** |
| **비동기 (Asynchronous)** | 명령어와 무관하게 외부에서 발생 → 언제 와도 처리 가능해야 |

---

## 3. 인터럽트 처리 흐름

```
1. 장치/명령어 → 인터럽트 신호 발생
2. CPU: 현재 명령어 완료 후 (atomic)
3. 현재 상태 저장 (PC, 레지스터, PSW)
4. 인터럽트 종류 확인 → IDT(Interrupt Descriptor Table)에서 핸들러 주소 찾기
5. 사용자 모드 → 커널 모드 전환
6. ISR(Interrupt Service Routine) 실행
7. 상태 복원 → 사용자 모드 복귀
8. 중단된 위치부터 재개
```

### 3.1 IDT (Interrupt Descriptor Table)
- x86의 인터럽트 핸들러 테이블 (256개 슬롯)
- 각 슬롯에 핸들러 주소
- 0~31: CPU 예외 (Divide Error, Page Fault, ...)
- 32~: 하드웨어 IRQ, 소프트웨어 인터럽트

### 3.2 PIC / APIC
- **PIC (Programmable Interrupt Controller)**: 옛날 단일 CPU용 (Intel 8259)
- **APIC (Advanced PIC)**: 멀티코어용. Local APIC + I/O APIC
- 각 장치의 IRQ 신호를 모아서 CPU에 전달

### 3.3 인터럽트 컨텍스트 vs 프로세스 컨텍스트
- **인터럽트 컨텍스트**에서는 sleep/block 불가, 너무 오래 머물면 안 됨
- 긴 작업은 **bottom half(softirq/tasklet/workqueue)**로 미룸
  - **Top half**: 즉시 처리할 최소한의 것 (예: 디바이스 ack)
  - **Bottom half**: 시간이 걸리는 작업 (예: 데이터 처리)

---

## 4. 인터럽트 우선순위 & 마스킹

### 4.1 우선순위
- 동시에 여러 인터럽트가 발생할 수 있음
- 우선순위가 더 높은 인터럽트가 낮은 것을 선점할 수 있음
- 같은 우선순위는 보통 순차 처리

### 4.2 마스킹 (Masking)
- 일시적으로 특정 인터럽트를 차단
- **Maskable Interrupt**: 차단 가능 (보통 하드웨어 IRQ)
- **NMI (Non-Maskable Interrupt)**: 차단 불가 (메모리 패리티 오류, 워치독)

### 4.3 인터럽트 비활성화 주의
- 짧게만 비활성화 (커널 임계 구역 보호용)
- 길게 막으면 응답성↓ + 인터럽트 누락 가능

## 5. 시스템 콜 (System Call)

> 유저 프로세스가 OS 커널의 기능을 사용하기 위한 **공식 인터페이스**.

### 5.1 왜 시스템 콜이 필요한가?
- 유저 모드는 하드웨어를 직접 못 만짐 (보호)
- 파일 읽기, 네트워크 통신, 프로세스 생성 등은 모두 커널만 가능
- **유저 모드 → 커널 모드 전환**이 필요 → 시스템 콜이 이 통로

### 5.2 시스템 콜의 종류

| 카테고리 | 예시 (Linux) |
|----------|-------------|
| 프로세스 제어 | `fork`, `exec`, `exit`, `wait` |
| 파일 관리 | `open`, `read`, `write`, `close`, `lseek` |
| 디바이스 관리 | `ioctl`, `read`, `write` |
| 정보 유지 | `getpid`, `gettimeofday` |
| 통신 | `pipe`, `socket`, `send`, `recv` |
| 보호 | `chmod`, `umask` |

### 5.3 시스템 콜 호출 흐름

```
1. 라이브러리 함수 호출 (예: glibc의 read())
2. 시스템 콜 번호를 레지스터에 세팅
3. 트랩 명령어 실행 (x86-64: syscall, ARM: svc)
4. CPU: 사용자 모드 → 커널 모드, IDT/syscall table 참조
5. 커널의 sys_read() 핸들러 실행
6. 결과를 레지스터에 세팅
7. 커널 모드 → 사용자 모드 복귀
8. 라이브러리 함수가 결과 반환
```

### 5.4 인터럽트 vs 시스템 콜
**둘 다 모드 전환을 일으키지만 본질은 다르다.**

| 항목 | 인터럽트 | 시스템 콜 |
|------|----------|-----------|
| **발생 주체** | 주로 외부 장치 (비동기) | 유저 프로세스 (의도적) |
| **목적** | 이벤트 통지 | 커널 기능 요청 |
| **동기성** | 비동기 (하드웨어) | 동기 (소프트웨어 트랩) |
| **재실행** | 발생 시점 명령어 그대로 재개 | 호출 명령어 다음으로 진행 |

→ 시스템 콜은 **"의도적으로 일으키는 동기 인터럽트(트랩)"**라고 볼 수 있다.

---

## 6. 사용자 모드 vs 커널 모드

### 6.1 CPU의 보호 링 (Protection Rings)
- **Ring 0 (커널 모드)**: 모든 명령어 실행 가능
- **Ring 3 (사용자 모드)**: 제한된 명령어만, 메모리 접근 제한
- (x86은 Ring 0~3 있지만 보통 0과 3만 사용)

### 6.2 특권 명령어 (Privileged Instructions)
- I/O 명령, 페이지 테이블 조작, 인터럽트 마스킹 등
- 유저 모드에서 시도 → **General Protection Fault → SIGSEGV**

### 6.3 모드 전환 시점
- 시스템 콜 호출 (트랩)
- 인터럽트 발생
- 예외 발생

→ **유저 모드는 커널 모드로 마음대로 못 들어가고, 정해진 입구(트랩/인터럽트)로만 가능**.

## 7. 컨텍스트 스위칭과의 관계

인터럽트가 발생한다고 항상 컨텍스트 스위칭이 일어나는 건 아니다.

| 시나리오 | 컨텍스트 스위칭 |
|----------|----------------|
| 짧은 인터럽트 처리 후 같은 프로세스 재개 | ❌ (모드 전환만) |
| 타이머 인터럽트 → 스케줄러 → 다른 프로세스 선택 | ✅ (선점 발생) |
| I/O 완료 인터럽트 → 대기 중이던 고우선순위 프로세스 깨움 | ✅ 가능 |
| 시스템 콜이 블로킹 (예: `read`에서 대기) | ✅ 다른 프로세스에 CPU 양도 |

**모드 전환 ≠ 컨텍스트 스위칭**
- 모드 전환: 사용자 ↔ 커널 (같은 프로세스 내에서도 일어남)
- 컨텍스트 스위칭: 프로세스/스레드 A → B 교체

## 8. 실무에서 마주치는 인터럽트/시스템 콜

### 8.1 strace
시스템 콜을 실시간으로 추적:
```bash
strace -c ls       # ls가 어떤 시스템 콜을 얼마나 부르는지 통계
strace -p PID      # 실행 중 프로세스 추적
```

### 8.2 perf / ftrace
- 인터럽트, 시스템 콜, 컨텍스트 스위칭 등의 빈도/지연 분석
- 성능 튜닝의 1순위 도구

### 8.3 시스템 콜 비용
- 일반 함수 호출: ~ns
- 시스템 콜: ~수십 ns ~ μs
- 그래서 **버퍼링** (`fread`가 매번 `read` 안 부르는 이유)
- **vDSO**: `gettimeofday`처럼 자주 부르는 건 시스템 콜 없이 처리 (x86-64에서 사용자 공간에 매핑)

### 8.4 인터럽트 폭주
- 1Gbps 이상 NIC는 패킷마다 인터럽트면 CPU가 죽음
- **NAPI (New API)**: 첫 인터럽트 후엔 폴링으로 전환 → 둘의 장점 결합
- **인터럽트 코얼레싱(Coalescing)**: 여러 이벤트를 모아서 한 번에 통지

## 9. 면접 단골 질문

### Q1. 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이는?
- **하드웨어**: 외부 장치, 비동기. 키보드/타이머/디스크
- **소프트웨어 (Trap)**: 명령어 실행 중 발생, 동기. 시스템 콜, 예외

### Q2. 인터럽트와 시스템 콜의 차이는?
→ §5.4 표 참조. **시스템 콜 = 의도적으로 일으키는 동기 트랩.**

### Q3. 인터럽트 처리 중 또 인터럽트가 오면?
- 일반적으로 우선순위가 더 높으면 **중첩(Nested Interrupt)** 가능
- 같거나 낮으면 마스킹되었다가 나중에 처리
- 동일 IRQ는 보통 마스킹

### Q4. 사용자 모드에서 커널 메모리에 직접 접근하면?
- **General Protection Fault**가 발생 → 커널이 SIGSEGV로 프로세스 종료
- 페이지 권한 비트(supervisor) 위반

### Q5. 시스템 콜이 함수 호출보다 비싼 이유는?
- 모드 전환 비용 (레지스터 저장/복원, TLB 영향 가능)
- 커널 진입/탈출 검사 (보안 검사)
- 캐시 미스 가능성
- 그래서 **시스템 콜 횟수를 줄이는 게 최적화 포인트** (예: 버퍼링, batch I/O)

### Q6. 페이지 폴트는 인터럽트인가?
- **Fault (소프트웨어 인터럽트의 한 종류)**.
- 페이지 폴트 발생 → 핸들러가 페이지 적재 → 발생 명령어 **재실행**.

### Q7. Top half / Bottom half가 무엇인가?
- **Top half**: 인터럽트 즉시 처리. 최소한 (장치 응답, 데이터 복사 정도). 인터럽트 비활성 상태일 수도 있어서 짧아야 함
- **Bottom half**: 시간이 걸리는 작업을 나중에 처리 (softirq, tasklet, workqueue)
- 응답성 + 처리량을 동시에 잡는 패턴
