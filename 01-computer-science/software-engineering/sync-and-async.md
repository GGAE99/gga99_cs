# 동기(Synchronous)와 비동기(Asynchronous)

> **한 줄 정리**
> **동기**: 호출자가 결과가 나올 때까지 **그 자리에서 기다린다**. 순서가 보장된다.
> **비동기**: 호출자가 결과를 기다리지 않고 **다른 일을 한다**. 결과는 나중에 콜백/이벤트/await로 받는다.
>
> 핵심은 **"호출자가 다음 줄로 넘어가서 다른 일을 할 수 있는가"**.

---

## 1. 왜 이렇게 헷갈리는가?

대부분 사람이 헷갈리는 이유는 **두 개의 다른 축**이 섞여있기 때문이다.

### 1.1 두 개의 축
- **동기 / 비동기 (Synchronous / Asynchronous)** — **결과 처리 시점**의 문제
- **블로킹 / 논블로킹 (Blocking / Non-blocking)** — **제어권 반환 시점**의 문제

언뜻 같아 보이지만 다르다. **이 둘을 분리해서 이해하면 거의 다 풀린다**.

### 1.2 어떻게 다른가?

| 축 | 질문 |
|----|------|
| 동기 vs 비동기 | "결과를 누가 신경 쓰나? 호출자가 직접 기다리나, 시스템이 알려주나?" |
| 블로킹 vs 논블로킹 | "함수가 곧장 리턴하나, 작업 끝날 때까지 멈춰있나?" |

---

## 2. 정의

### 2.1 동기 (Synchronous)
- 호출자가 **결과가 나올 때까지 그 자리에서 대기**
- 호출 순서 = 완료 순서
- 코드 흐름이 단순, 디버깅 쉬움

```js
const result = readFileSync("a.txt");  // 끝날 때까지 멈춤
console.log(result);                   // 결과 나온 후 실행
```

### 2.2 비동기 (Asynchronous)
- 호출자가 **결과를 기다리지 않고 다음 일로 진행**
- 결과는 콜백 / Promise / 이벤트 / await 등으로 나중에 받음
- 코드 흐름 복잡, 동시성 ↑

```js
readFile("a.txt", (err, data) => {  // 완료 시 호출될 콜백 등록
    console.log(data);
});
console.log("기다리지 않고 계속");   // 위 콜백보다 먼저 출력될 수 있음
```

### 2.3 블로킹 (Blocking)
- 호출이 **즉시 리턴하지 않음**. 작업이 끝나야 다음 줄 실행
- CPU를 점유한 채 기다리거나, 커널이 스레드를 sleep 상태로 둠

### 2.4 논블로킹 (Non-blocking)
- 호출이 **즉시 리턴**
- 작업이 끝나지 않았으면 "아직 안 됨" 같은 응답 (EAGAIN, EWOULDBLOCK 등)
- 호출자는 나중에 다시 확인하거나, 다른 메커니즘으로 결과 받음

### 2.5 네 가지 조합

| | 블로킹 | 논블로킹 |
|---|--------|---------|
| **동기** | **Sync + Blocking** <br>일반적인 `read()`, `fetch()` await | **Sync + Non-blocking** <br>`O_NONBLOCK` + 폴링 |
| **비동기** | **Async + Blocking** <br>`select`/`poll`/`epoll` (논쟁 있음) | **Async + Non-blocking** <br>Node.js 콜백, io_uring, AIO |

**가장 자주 보는 두 가지**
1. **동기 + 블로킹**: "함수 호출하고 결과 받을 때까지 멈춤" — 일반적
2. **비동기 + 논블로킹**: "요청만 던지고 결과는 나중에" — Node.js, 이벤트 루프

## 3. 일상 비유로 이해하기

### 3.1 카페 주문 시나리오

**동기 + 블로킹**
> 커피 주문 후 **카운터 앞에 가만히 서서** 커피 나올 때까지 기다린다.
> → 손님(호출자)은 다른 일을 못 함.

**비동기 + 논블로킹**
> 커피 주문하고 **진동벨 받고 자리로 돌아가서 일을 본다**. 벨이 울리면 가지러 감.
> → 손님은 그동안 다른 일을 할 수 있음.

**동기 + 논블로킹** (어색하지만 가능)
> 주문하고 **자리로 돌아가지만, 매분마다 카운터로 가서 "다 됐어요?" 묻는다**.
> → 손님은 결국 본인이 계속 신경 써야 함. 효율 안 좋음.

**비동기 + 블로킹** (이상한 조합)
> 진동벨을 받았는데도 카운터 앞에 계속 서 있는 셈. 시스템은 비동기인데 호출자가 굳이 기다림.

### 3.2 핵심 비유 요약

| 측면 | 동기 | 비동기 |
|------|------|--------|
| **누가 완료를 확인하나?** | 호출자가 직접 | 시스템이 통지 (콜백/이벤트) |
| **호출자의 멘탈 모델** | "내가 기다린다" | "끝나면 알려줘" |

| 측면 | 블로킹 | 논블로킹 |
|------|--------|---------|
| **함수 리턴 시점** | 작업 끝난 후 | 즉시 (결과는 아직 없음) |

## 4. 실제 코드 예시

### 4.1 파일 읽기

```js
// 동기 + 블로킹
const data = fs.readFileSync("a.txt");      // 끝날 때까지 멈춤
process(data);

// 비동기 + 논블로킹 (콜백)
fs.readFile("a.txt", (err, data) => {       // 즉시 리턴, 완료 시 콜백
    process(data);
});

// 비동기 + 논블로킹 (async/await)
const data = await fs.promises.readFile("a.txt");  // 비동기지만 코드는 동기처럼 보임
process(data);
```

> `async/await`는 **비동기지만 코드 모양은 동기처럼** 보이게 해주는 문법 설탕. 실제 동작은 비동기.

### 4.2 네트워크 요청

```python
# 동기
response = requests.get("https://api.example.com")
print(response.json())          # 응답 받기 전엔 멈춤

# 비동기 (asyncio + aiohttp)
async with aiohttp.ClientSession() as session:
    async with session.get("https://api.example.com") as resp:
        print(await resp.json())  # await 동안 다른 코루틴 실행 가능
```

### 4.3 시스템 콜 레벨

```c
// 동기 + 블로킹 (기본)
int n = read(fd, buf, 1024);    // 데이터 올 때까지 스레드 sleep

// 동기 + 논블로킹
fcntl(fd, F_SETFL, O_NONBLOCK);
int n = read(fd, buf, 1024);    // 데이터 없으면 -1 + EAGAIN 즉시 리턴

// 비동기 (Linux AIO / io_uring)
io_uring_prep_read(...);         // 요청만 큐에 넣음
// ... 다른 일 하다가 ...
io_uring_wait_cqe(...);          // 완료 알림 수신
```

## 5. 언제 동기를 쓰고 언제 비동기를 쓰나? (가장 중요)

> **헷갈릴 때 떠올릴 단 하나의 질문:**
> **"이 작업이 끝날 때까지 기다리는 동안, 호출자가 할 수 있는 다른 일이 있는가?"**
> - 없거나 의미 없으면 → **동기**
> - 있고 가치 있으면 → **비동기**

### 5.1 동기를 쓰는 게 좋은 경우

| 상황 | 이유 |
|------|------|
| **결과 없이는 다음 줄을 못 쓰는 경우** | 어차피 기다려야 함. 비동기 복잡도만 더해짐 |
| **순서가 절대적으로 중요한 트랜잭션** | DB 트랜잭션 내 SQL, 결제 처리의 검증→차감→로그 |
| **CPU 바운드 짧은 작업** | I/O 대기가 없으니 비동기 이득 없음 |
| **앱 시작 시 설정 로드** | 어차피 끝나야 진행 가능. 단순함 우선 |
| **테스트 코드** | 순서/타이밍 명확해야 검증 쉬움 |
| **CLI 도구의 단발성 작업** | `cp`, `mv` 같은 명령. 비동기 의미 없음 |

### 5.2 비동기를 쓰는 게 좋은 경우

| 상황 | 이유 |
|------|------|
| **I/O 대기가 길어 다른 일 할 수 있을 때** | 네트워크/디스크 대기 동안 다른 요청 처리 |
| **UI 응답성을 지켜야 할 때** | 메인 스레드가 멈추면 화면 얼어붙음 |
| **수많은 동시 커넥션 처리 (C10K)** | 스레드당 메모리/스위칭 비용 ↑. 이벤트 루프로 절약 |
| **여러 독립적인 작업을 병렬 진행** | `Promise.all`처럼 여러 API 동시 호출 |
| **이벤트/메시지 기반 시스템** | 메시지 큐, 알림 시스템, 채팅 등 |
| **백그라운드 작업** | 이메일 발송, 썸네일 생성, 인덱싱 |

### 5.3 결정 트리

```
이 작업의 결과가 즉시 필요한가?
├─ YES → 동기로 작성 (await 포함)
└─ NO → 작업 시간이 긴가?
        ├─ NO (수 ms 이내) → 동기로도 OK
        └─ YES → 그동안 호출자가 할 일이 있는가?
                ├─ NO → 어차피 기다리니 동기
                └─ YES → 비동기 (콜백/Promise/이벤트)
```

### 5.4 흔한 실수

**❌ 무조건 비동기**
- `async/await`를 다 붙였지만 실제 동작은 직렬 — 비동기 이득 없이 복잡도만 ↑
- 짧은 CPU 작업까지 비동기로 → 컨텍스트 스위칭 오버헤드만 늘어남

**❌ 비동기 코드에서 동기 함수 호출**
- Node.js 이벤트 루프에서 `readFileSync`를 호출하면 **전체 이벤트 루프가 멈춤** → 모든 요청 응답 지연
- 비동기 환경에선 반드시 비동기 API 사용

**❌ 비동기 API를 동기처럼 사용**
- Promise를 await 없이 호출하고 결과를 그냥 쓰려고 하기
- `.then` 콜백 안에 또 `.then`을 중첩해서 콜백 지옥

### 5.5 실무 가이드 요약

> - **단순한 스크립트, CLI, 짧은 작업**: 동기로 시작. 충분히 빠르면 그대로.
> - **서버**: 거의 모든 I/O는 비동기. CPU 바운드만 별도 스레드/프로세스로.
> - **UI**: I/O와 무거운 계산은 비동기 (UI 스레드 보호).
> - **DB 트랜잭션 내부**: 순서가 중요하므로 동기.

## 6. 프로세스 / 스레드와의 관계

→ 본격적으로 OS 레벨에서 비동기가 왜 의미 있는지.

### 6.1 동기 + 멀티스레드 (전통적 서버 모델)

```
서버
├─ 스레드 1 → 요청 A 처리 (DB 조회 동안 sleep)
├─ 스레드 2 → 요청 B 처리 (파일 읽기 동안 sleep)
├─ 스레드 3 → 요청 C 처리 (네트워크 응답 대기)
└─ ...
```

- 각 스레드는 **동기적으로** 일하지만, **여러 스레드**가 있어서 시스템 전체는 동시성 확보
- **장점**: 코드 단순. 동기 코드 그대로 작성
- **단점**:
  - 스레드당 메모리(스택) 수 MB ~ 수십 MB
  - 스레드 ↑ → 컨텍스트 스위칭 비용 폭증
  - **C10K 문제**: 1만 동시 커넥션 시 1만 스레드 = 비현실적

→ 대표 예: 전통적인 Spring Boot + Tomcat (스레드 풀 200~500개로 동기 처리)

### 6.2 비동기 + 싱글스레드 (이벤트 루프 모델)

```
서버 (단일 스레드)
└─ Event Loop
    ├─ 요청 A → DB 요청 보내고 즉시 다음
    ├─ 요청 B → 파일 요청 보내고 즉시 다음
    ├─ 요청 C → ...
    └─ 어느 요청이 완료되면 콜백 실행 (epoll/kqueue로 감지)
```

- 스레드는 **한 개**, I/O는 **비동기 + 논블로킹**
- 스레드는 절대 I/O에서 sleep하지 않음. 항상 "할 일"을 찾아 다님
- **장점**:
  - 메모리 효율적 (스레드 수 적음)
  - 컨텍스트 스위칭 거의 없음
  - 수만 동시 커넥션 처리 가능
- **단점**:
  - CPU 바운드 작업 한 번이라도 길게 돌면 **전체 멈춤**
  - 코드 복잡 (콜백, async/await 체인)
  - 에러 추적 어려움

→ 대표 예: Node.js, Nginx, Redis

### 6.3 하이브리드 (현실적인 접근)

대부분의 모던 시스템은 **둘을 혼합**:

```
메인 이벤트 루프 (비동기 I/O)
        ↓ CPU 바운드 작업 발견
   워커 스레드 풀로 위임 (동기 처리)
        ↓ 결과
   이벤트 루프가 콜백 호출
```

**예시**
- **Node.js**: 이벤트 루프(단일) + libuv 스레드 풀(파일 I/O, DNS 등 일부 시스템 콜용)
- **Nginx**: 마스터 + 워커 프로세스 (각 워커는 이벤트 루프)
- **Python asyncio**: 이벤트 루프 + `run_in_executor`로 동기 코드 위임
- **Java**: 가상 스레드(Project Loom, Java 21) — 동기 코드 작성, 내부적으로 비동기 처리

### 6.4 비동기는 어떻게 스레드를 절약하나?

**동기 모델 (스레드 N개)**
- 스레드 100개 × 8MB 스택 = 800MB
- 99%가 I/O 대기 중 → 메모리만 차지하고 놀고 있음

**비동기 모델 (스레드 1개)**
- 스레드 1개로 모든 I/O를 epoll로 한 번에 감시
- 어느 fd가 준비되면 그 콜백 실행
- 메모리는 콜백 수만큼 (수 KB)

→ **I/O 바운드 워크로드에서 비동기가 압도적으로 효율적**.
→ **CPU 바운드면 비동기 의미 없음** (어차피 한 스레드는 한 번에 하나만 계산).

### 6.5 동기-비동기 vs 멀티스레드 정리

| 모델 | 동시성 방식 | 적합한 워크로드 | 예시 |
|------|-----------|---------------|------|
| **동기 + 싱글스레드** | 없음 | 단순 스크립트, CLI | `cat`, `cp`, Python 스크립트 |
| **동기 + 멀티스레드** | 스레드 풀로 병렬 | I/O 적당, CPU 적당 | Tomcat, 전통적 서버 |
| **비동기 + 싱글스레드** | 이벤트 루프 | I/O 매우 많음, CPU 적음 | Node.js, Nginx, Redis |
| **비동기 + 멀티스레드** | 둘 다 | 거의 모든 모던 서버 | Netty, Tokio, Go runtime |

## 7. 자주 헷갈리는 포인트

### 7.1 `await`는 블로킹인가?
**No.** `await`는 **현재 코루틴**을 대기시킬 뿐, **스레드는 다른 일을 함**.
```python
async def fetch():
    data = await http_get(url)  # 이 코루틴은 멈추지만, 이벤트 루프는 다른 코루틴 실행
    return data
```

### 7.2 멀티스레드면 자동으로 비동기인가?
**No.** 멀티스레드도 **각 스레드 안에서는 동기적**으로 실행 가능. "여러 스레드를 동시에 돌린다 = 동시성"이지 "비동기"는 아님.

### 7.3 비동기는 항상 빠른가?
**No.**
- 단일 짧은 작업: 동기가 더 빠름 (오버헤드 없음)
- CPU 바운드: 비동기 의미 없음 (병렬화는 멀티프로세싱/멀티스레딩의 영역)
- I/O 바운드 + 동시 요청 多: 비동기가 압도적

### 7.4 동기 코드도 동시성을 가질 수 있다
멀티스레드 + 동기 코드 = 동시성 O, 비동기성 X.
"동기 ≠ 단일 실행"이라는 점을 명확히.

### 7.5 비동기는 병렬과 다르다
- **비동기**: 한 일을 던져놓고 다른 일을 함 (반드시 동시 실행은 아님)
- **병렬**: 정말 동시에 여러 일을 실행 (멀티코어 활용)
- → 싱글코어에서도 비동기 가능, 병렬은 멀티코어 필요

## 8. 면접 단골 질문

### Q1. 동기와 비동기, 블로킹과 논블로킹의 차이?
- **동기/비동기**: 결과를 누가 신경 쓰나 (호출자 vs 시스템)
- **블로킹/논블로킹**: 함수가 즉시 리턴하는가
- 두 개의 다른 축 → 네 가지 조합 (§2.5)

### Q2. 비동기가 항상 좋은가?
**No.**
- I/O 바운드 + 동시성 필요 → 비동기 유리
- CPU 바운드 → 비동기 이점 없음, 오히려 복잡도만↑
- 짧은 단발성 → 동기가 단순하고 충분히 빠름

### Q3. Node.js는 어떻게 싱글스레드로 수만 커넥션을 처리하나?
- **이벤트 루프 + 비동기 논블로킹 I/O**
- 스레드는 I/O에서 sleep 안 함 → 항상 일하는 상태
- epoll/kqueue로 모든 fd를 한 번에 감시
- 단, **CPU 바운드 작업이 들어오면 전체 멈춤** → 워커 스레드/클러스터로 분리

### Q4. async/await는 멀티스레딩과 같은가?
**No.**
- async/await는 **싱글 스레드 안에서 비동기를 다루는 문법**
- 코루틴이 await 만나면 양보 → 이벤트 루프가 다른 코루틴 실행
- 진정한 병렬 실행이 필요하면 멀티스레드/멀티프로세스 별도 필요

### Q5. 비동기 코드에서 동기 함수를 호출하면 어떻게 되나?
- **이벤트 루프가 멈춤** → 모든 다른 비동기 작업이 지연
- Node.js에서 `readFileSync`, `executeSync` 같은 함수 호출은 절대 금지
- 어쩔 수 없이 동기 작업을 해야 하면 **워커 스레드 풀로 위임**

### Q6. 멀티스레드 동기 vs 싱글스레드 비동기, 언제 어느 걸?
- **CPU 바운드 / 작업이 무거움** → 멀티스레드 (병렬 처리)
- **I/O 바운드 / 동시 커넥션 많음** → 싱글스레드 비동기 (메모리·스위칭 비용 절감)
- **둘 다 필요** → 하이브리드 (Node.js의 cluster, Go의 goroutine, Java 가상 스레드)

### Q7. Go의 고루틴이나 Java 가상 스레드는 동기인가 비동기인가?
- **코드는 동기처럼 보이지만, 런타임이 내부적으로 비동기 + 스케줄링**을 처리
- 개발자는 동기 코드 작성, 런타임이 I/O 대기 시 다른 고루틴/가상 스레드 실행
- 동기 코드의 단순함 + 비동기의 성능을 결합한 모델
