## Q. URL과 URI에 대해 설명하시오.

??? note "정답 보기 ▼"
    - [x] URI = URL ∪ URN — URL은 위치(어디서 가져오는지), URN은 이름(무엇인지)을 의미하며 모든 URL은 URI에 포함된다
    - [ ] URI 안에서 `?` 뒤에 위치해 키-값 쌍으로 자원 변형 조건을 전달하는 부분이 쿼리 스트링(query string)이다
    - [ ] URI에서 `#` 뒤에 위치해 같은 문서 내 특정 위치를 가리키는 부분이 프래그먼트 식별자(fragment identifier)다
    - [ ] 비ASCII 문자나 예약 문자를 `%XX` 형태로 바꾸어 전송 안전성을 확보하는 것이 퍼센트 인코딩(URL encoding)이다

### 핵심 답안 (30초)
> _URI = URL ∪ URN. URL은 위치, URN은 이름. 모든 URL은 URI._

## Q. TCP와 UDP에 대해 설명하시오.

??? note "정답 보기 ▼"
    - [x] TCP는 연결지향·신뢰성·순서 보장, UDP는 비연결·빠름·손실 허용 — 둘 다 OSI 4계층(전송) 프로토콜
    - [ ] QUIC는 UDP 위에서 TCP의 신뢰성과 TLS 핸드셰이크를 통합한 프로토콜로 HTTP/3의 기반이 된다
    - [ ] SCTP는 메시지 지향과 멀티 스트림을 지원하는 4계층 프로토콜로, TCP의 head-of-line blocking 문제를 완화한다
    - [ ] DCCP는 비신뢰 데이터그램에 혼잡 제어를 더한 프로토콜로 미디어 스트리밍 같은 워크로드를 겨냥해 설계되었다

### 핵심 답안 (30초)
> _TCP: 연결지향·신뢰성·순서 보장. UDP: 비연결·빠름·손실 허용._

## Q. TCP 3-way handshake에 대해 설명하시오.

??? note "정답 보기 ▼"
    - [x] SYN → SYN-ACK → ACK 순서로 양쪽이 시퀀스 번호를 교환·확인하며 연결을 수립
    - [ ] 종료 시 FIN → ACK → FIN → ACK 4단계를 거치며 양방향을 독립적으로 닫는 4-way handshake 절차
    - [ ] 능동 닫기 측이 RTT 보호와 늦은 패킷 처리 안전을 위해 일정 시간 머무는 TIME_WAIT 상태
    - [ ] 초기에 SYN+ACK 데이터를 한꺼번에 보내 1-RTT를 절약하는 TCP Fast Open(TFO) 확장

### 핵심 답안 (30초)
> _SYN → SYN-ACK → ACK. 시퀀스 번호 동기화와 연결 수립._

## Q. HTTP의 주요 특성 2가지를 말하고 설명하시오.

??? note "정답 보기 ▼"
    - [x] Stateless(상태 없음) + Connectionless(원래 매 요청마다 연결 종료) — 확장성 ↑, 상태는 쿠키/세션/토큰으로 보완
    - [ ] HTTP/1.1의 Keep-Alive로 같은 TCP 연결을 재사용해 RTT를 줄이는 Persistent Connection 특성
    - [ ] HTTP/2의 하나의 연결에서 다수 요청을 동시에 다루는 멀티플렉싱(스트림) + 바이너리 프레이밍 특성
    - [ ] HTTP/3의 QUIC 기반 0-RTT 재연결과 패킷 손실이 한 스트림에만 영향을 주는 독립 스트림 특성

### 핵심 답안 (30초)
> _Stateless + Connectionless(원래). 확장성 ↑, 그러나 상태 유지 필요시 쿠키/세션/토큰._

## Q. 웹소켓과 HTTP의 차이점을 설명하시오.

??? note "정답 보기 ▼"
    - [x] HTTP는 요청-응답 단방향, WebSocket은 양방향 풀 듀플렉스이며 초기 핸드셰이크만 HTTP Upgrade를 사용
    - [ ] SSE(Server-Sent Events)는 HTTP 한 연결을 열어 두고 서버가 단방향으로 텍스트 이벤트를 푸시하는 방식이다
    - [ ] Long polling은 서버가 응답을 일정 시간 보류했다가 데이터가 생기면 응답하고 클라이언트가 즉시 재요청하는 방식이다
    - [ ] HTTP/2 Server Push는 같은 연결 위에서 클라이언트가 요청하지 않은 자원을 미리 푸시하는 기능으로, 양방향 메시징과는 다르다

### 핵심 답안 (30초)
> _HTTP는 요청-응답 단방향. WebSocket은 양방향 풀 듀플렉스. 초기 핸드셰이크는 HTTP Upgrade._

## Q. NAT(Network Address Translation)에 대해 설명하시오.

??? note "정답 보기 ▼"
    - [x] 사설 IP와 공인 IP 사이의 주소 변환 — IPv4 주소 부족 완화 + 내부 망 은닉 효과
    - [ ] 외부 포트를 내부 특정 호스트·포트로 매핑해 인바운드 연결을 가능하게 하는 PAT(포트 포워딩)
    - [ ] 같은 LAN 안에서 IP 주소를 MAC 주소로 매핑해 프레임 전송 대상을 알아내는 ARP 프로토콜
    - [ ] NAT 환경에서 P2P 직접 연결을 위해 양쪽이 같은 외부 매핑을 동시에 뚫는 NAT Traversal(STUN/TURN) 기법

### 핵심 답안 (30초)
> _사설 IP ↔ 공인 IP 변환. IPv4 주소 부족 완화 + 내부 망 은닉._

## Q. DNS는 무엇인가?

??? note "정답 보기 ▼"
    - [x] 도메인 ↔ IP 매핑을 제공하는 계층적 분산 시스템 — 재귀/반복 질의와 캐시 계층을 사용
    - [ ] A는 도메인 → IPv4, AAAA는 도메인 → IPv6, CNAME은 도메인 → 다른 도메인의 별칭을 정의하는 레코드 타입
    - [ ] MX는 메일 라우팅 대상을 우선순위와 함께 지정하고, TXT는 SPF/DKIM 같은 임의 문자열을 담아 메일 인증에 활용
    - [ ] 캐시 결과를 가짜로 주입해 클라이언트를 다른 서버로 유도하는 공격 기법인 DNS 캐시 포이즈닝(cache poisoning)

### 핵심 답안 (30초)
> _도메인 ↔ IP 매핑. 계층적 분산 시스템. 재귀/반복 질의, 캐시 계층._

### 꼬리 질문 후보
- A / AAAA / CNAME / MX / TXT 레코드의 차이는?
- DNS 캐시 포이즈닝이란?

## Q. CDN이란 무엇인가?

??? note "정답 보기 ▼"
    - [x] 전 세계 엣지 서버에 콘텐츠를 캐싱해 지연을 줄이고 원본 서버 부하를 분산하는 네트워크
    - [ ] 사용자 요청을 가장 가까운 엣지 PoP로 보내기 위해 Anycast IP와 GeoDNS로 라우팅하는 트래픽 분산 기법
    - [ ] 원본 서버 부담을 줄이기 위해 미들 티어에서 한 번 더 캐시를 모아 두는 Origin Shielding(Tiered Caching) 구조
    - [ ] 엣지에서 함수를 실행해 응답을 가공·인증·라우팅하는 Edge Functions / Edge Workers 같은 컴퓨팅 모델

### 핵심 답안 (30초)
> _전 세계 엣지 서버에 콘텐츠 캐싱. 지연 단축 + 원본 서버 부하 분산._
