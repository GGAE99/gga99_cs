## Q. 프로세스 동기화 문제와 뮤텍스·세마포어의 차이를 설명하시오.

??? note "정답 보기 ▼"
    - [x] 경쟁 조건 → 임계 구역 보호. Mutex는 소유 개념의 이진 락, Semaphore는 카운팅 기반 신호
    - [ ] 데드락의 4가지 필요 조건은 상호 배제, 점유 대기, 비선점, 순환 대기이며 하나라도 깨면 데드락이 발생하지 않는다
    - [ ] 모니터(monitor)는 락과 조건 변수를 한 객체에 묶어 자동으로 진입·해제를 처리하는 고수준 동기화 추상화다
    - [ ] 스핀락은 짧은 임계구역에서 sleep 대신 바쁜 대기를 도는 락으로 컨텍스트 스위칭 비용을 피하려는 목적의 동기화 기법이다

### 핵심 답안 (30초)
> _경쟁 조건 → 임계 구역 보호. Mutex: 소유 개념의 이진 락. Semaphore: 카운팅 신호._

## Q. 뮤텍스 락에 대해 간략하게 설명하시오.

??? note "정답 보기 ▼"
    - [x] 상호 배제(Mutual Exclusion) — 락을 획득한 스레드만 해제 가능한 소유권 개념의 이진 락
    - [ ] 짧은 임계구역에서 sleep 대신 CPU를 점유하며 락 획득을 기다리는 spinlock — 컨텍스트 스위칭 비용 회피용
    - [ ] 같은 스레드가 자신이 들고 있는 락을 중첩 획득할 수 있도록 카운팅을 추가한 재귀(reentrant) 뮤텍스
    - [ ] 임계구역 대신 데이터 자체를 원자적으로 갱신해 락을 쓰지 않는 lock-free / CAS 기반 동기화 기법

### 핵심 답안 (30초)
> _상호 배제. 락을 획득한 스레드만 해제 가능. 소유권 개념._

## Q. 세마포어에 대해 간략하게 설명하시오.

??? note "정답 보기 ▼"
    - [x] 정수 카운터 기반 동기화 — wait(P)/signal(V)로 자원 풀을 관리하기에 적합
    - [ ] 카운터를 0/1로 제한해 사실상 이진 락처럼 쓰는 형태인 이진 세마포어(binary semaphore)이며 소유 개념이 없다는 점에서 mutex와 구별된다
    - [ ] 락과 조건 변수를 한 객체로 묶어 진입·신호 전파를 일관되게 처리하는 고수준 동기화 추상화인 모니터(monitor)
    - [ ] 락 없이 원자적 비교 후 교환(CAS)으로 공유 변수를 갱신하는 lock-free 기법으로, 카운팅 동기화와는 다른 접근

### 핵심 답안 (30초)
> _정수 카운터 기반 동기화. wait(P) / signal(V). 자원 풀 관리에 적합._

## Q. 외부 단편화에 대해 설명하시오.

??? note "정답 보기 ▼"
    - [x] 가용 메모리 총량은 충분하지만 연속된 블록이 없어 할당이 실패하는 현상 — 가변 분할 방식의 고유 문제
    - [ ] 고정 크기 블록에 작은 데이터를 담을 때 블록 안쪽에 남는 사용되지 않는 공간을 가리키는 내부 단편화(internal fragmentation)

    - [ ] 페이지 테이블이 너무 커져 TLB 미스가 잦아지고 가상 주소 변환 비용이 늘어나는 TLB 압박 현상

### 핵심 답안 (30초)
> _가용 메모리 총량은 충분하지만, 연속된 블록이 없어서 할당 실패하는 현상._

## Q. 스레드, 프로세스, 코어의 수는 많을수록 좋은가?

??? note "정답 보기 ▼"
    - [x] 아니다 — 컨텍스트 스위칭 비용, 동기화 오버헤드, Amdahl's Law 때문이며 워크로드 특성에 따라 결정해야 한다
    - [ ] CPU 바운드는 보통 (코어 수) 또는 (코어 수 + 1) 정도가 최적이고, 그 이상은 스위칭 비용만 늘어나 처리량이 떨어진다
    - [ ] I/O 바운드는 블로킹 대기 시간을 다른 스레드가 메우므로 코어 수의 수배까지 늘려도 유효한 작업이 늘어나는 경우가 많다
    - [ ] Gustafson's Law는 시스템 크기와 함께 문제 크기도 늘 때 더 낙관적인 스케일링을 예측하지만, 직렬 구간이 큰 워크로드에는 적용되지 않는다

### 핵심 답안 (30초)
> _NO. 컨텍스트 스위칭 비용, 동기화 오버헤드, Amdahl's Law. 워크로드 종속._
